# 11 — 체감 속도 & UX: 스트리밍 대기 피드백 + 중지 + 접근성

> 근거형 법률 답변은 본질적으로 10~25초 걸린다(출력 토큰 多). 총 시간을 더 줄이는 건 한계가 있으니, **"기다리는 동안 멈춘 듯 보이지 않게"** 만드는 게 사용자 체감의 핵심이다.

## 현재 상태 (코드 대조 2026-06-30, `web/chat.html`)
- SSE 스트리밍 사용 중(`stream:true`, `getReader`). `token`/`done`/`error` 이벤트만 처리.
- ❗ 백엔드가 **가장 먼저 보내는 `meta` 이벤트(`used_context`=근거 목록)를 프론트가 무시**한다.
- ❗ 첫 토큰 도착 전(질의 재작성+검색 수 초, 콜드면 수십 초) 화면엔 **깜빡이는 커서만** → "멈춘 것 같다".
- 중지 버튼 없음. 스트리밍 답변에 대한 스크린리더 알림(aria-live) 없음.
- (참고) 광고검토 카드 재바인딩·교정본 렌더·인용 신호등은 이미 반영됨 → 이번 작업 대상 아님.

## 목표
대기 구간을 **단계별 상태**로 채우고, 긴 생성을 **중지**할 수 있게 하며, **접근성**을 보강한다.

## 대상 파일
- `web/chat.html` (단일 페이지) 만 수정.

## 작업 단계
1. **단계별 상태 표시(빈 화면 제거)**:
   - 전송 직후 버블에 "🔎 관련 법령·판례를 찾고 있어요…"(점 애니메이션).
   - `meta` 수신 시 "✍️ 답변을 작성하고 있어요 · 근거 N건 확보"(N = `used_context.length`).
   - 첫 `token` 도착 시 상태 제거 후 답변 텍스트로 전환.
2. **`meta` 이벤트 처리 추가** (현재 누락).
3. **중지 버튼**: `AbortController`로 `fetch` 취소. 생성 중엔 전송 버튼이 "중지"로 토글, 취소 시 버블에 "중지됨" 표기. 중지 후 다시 입력 가능 상태 복원.
4. **접근성**: 답변 버블 `aria-live="polite"`(+ 상태 텍스트도 읽히게), 버튼에 명확한 라벨.
5. **소소한 폴리시**: 첫 사용 시 "답변은 보통 10~20초 걸려요" 1회 힌트, 토큰마다 강제 `scrollIntoView` 완화(사용자가 위로 스크롤하면 따라가지 않기), 입력 비활성 시각 표시.

## 완료 기준 (DoD)
- 코드상 `meta`/`token`/`done`/`error` **4개 이벤트 모두 처리**.
- 정적 점검: HTML 태그 균형, `<script>` 문법 오류 없음(빌드 후 `/chat` 로드 시 콘솔 에러 0).
- (사용자 직접 테스트) 질문 → 대기 중 단계 상태가 보임 → 토큰 스트림 → 인용 카드. 긴 답변 중 "중지" 동작.

## 주의 (회귀 금지)
- 기존 인용 카드(`renderCites`)·마크다운(`mdToHtml`)·광고검토(`review`)·데모키(`authHeaders`) 동작 유지.
- 응답 스키마에 **실재하는 필드만** 사용(`meta.used_context`, `done.answer/citations`). 추정 필드 금지.
