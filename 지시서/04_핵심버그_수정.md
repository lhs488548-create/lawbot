# P4 — 핵심 버그 수정 (정식화 전 필수)

근거: `docs/lawbot_완성도_개선_정리.md` §2 + **실제 코드 대조 검증(2026-06-29) 완료.** 아래 위치·라인은 검증 시점 기준이며, 작업 전 현재 코드로 재확인한다. 각 항목에 **검증 판정**을 표기했다(일부 항목은 "이미 구현됨"이라 재작업 금지).

## #1 [법적·CONFIRMED] as_of_date 게이트가 기본값으로 무장되지 않음 — 최우선
- 정확한 실태: 게이트 **기계 자체는 정상**(`search/retriever.py`의 `build_filter`가 `effective_from <= as_of_date` 필터). 문제는 **기본값 None이 retriever까지 도달하지 않아 필터가 영영 안 걸리는 것.**
  - `api/main.py` `as_of_date` 기본 None → rag로 전달 → `search/rag.py`의 `_call_search`가 **`as_of_date is not None`일 때만** 전달 → 기본 None이면 retriever에 아예 안 감 → 필터 객체 미생성 → **미래 시행 조문이 현행처럼 인용**.
  - `search/ad_review.py`는 `as_of_date`를 **아예 안 넘김**(별도 수정 필요).
- 수정: api·rag의 기본값을 **서버 오늘 날짜**로. `_call_search`의 `if as_of_date is not None` 가드 때문에 **api/rag 한쪽만 고치면 조용히 무효화**되니 둘 다. ad_review는 호출부에 as_of_date 추가. api의 TypeError 폴백(kwarg strip)은 retriever가 as_of_date를 받으므로 발동 안 함(확인).
- 파일: `api/main.py`, `search/rag.py`(`_call_search` 포함), `search/retriever.py`, `search/ad_review.py`.

## #2 [재분류 — 현재는 정상, P2 대비 주의] MIN_RETRIEVAL_SCORE 게이트
- 검증: `config.py` `MIN_RETRIEVAL_SCORE=0.20`. 게이트는 `search/rag.py`(일반·스트리밍 두 경로)에서 **dense 코사인 원점수(`hits[0].score`)에 적용 중.** 현재 retriever는 **dense 단독**(RRF/하이브리드 없음).
- 따라서 "RRF 점수(~0.016)가 게이트에 막혀 전 질의 차단"은 **현재 코드엔 해당 없음**(RRF가 아직 없음). 이미 "dense 원점수에만 적용"이라 그 부분은 충족됨.
- 할 일: (a) **0.20이 구어 질의에 과한지** 골든셋 top_score 분포로 재보정(P9). (b) **P2에서 하이브리드 추가 시, RRF 융합 점수로 게이트하지 말 것** — 게이트는 dense 원점수 유지, RRF는 정렬에만. (이건 "버그 수정"이 아니라 P2 작업 시 지켜야 할 제약.)
- 파일: `config.py`, `search/rag.py`(게이트 두 곳).

## #3 [높음·CONFIRMED] Citation Firewall 1차 DB 게이트 죽음
- 검증: `search/verify.py`가 제거된 `get_qdrant_client`를 import·호출(정의가 레포에 0개). try/except로 감싸 있어 **크래시는 아니고 항상 None 반환 → `parents.jsonl` 제목 조회로 degrade.** 즉 "항상 죽음"이 아니라 **Qdrant 경로만 죽고, 조문(article) 단위 존재확인은 사실상 불가, 제목 단위만 생존.**
- `api/main.py`도 `retriever.get_qdrant_client`를 참조(hasattr 가드라 healthz의 qdrant가 영구 false).
- 수정: FAISS 시대에 맞게 `metas`/`parents.jsonl`/source_url·case_no 조회로 조문 단위 존재확인 복구. api의 qdrant 참조도 정리(P8과 연동).
- 파일: `search/verify.py`, `api/main.py`.

## #4 [STALE — 이미 구현됨, 재작업 금지] 인덱스↔메타 정합성
- 검증: **이미 있음.** `embed/faiss_index.py` build에서 `assert index.ntotal == len(metas)`, 중복 chunk_id 거부, 차원 검증, L2 재정규화. load에서 `ntotal != len(metas)`면 RuntimeError. retriever도 row_id 범위 방어.
- 따라서 "정합성 검증을 추가하라"는 **불필요**(이미 통과). **유일한 보강거리**: build에서 add 직후 **벡터 L2노름 ≈ 1 사후검증**은 명시적으로 없으니 그것만 추가하면 됨(나머진 손대지 말 것).
- 파일: `embed/faiss_index.py`(L2노름 사후검증만).

## #5 [CONFIRMED 공백 / 위치 주의] trust_grade=B 미필터
- 검증: `embed/faiss_index.py` build_index는 src의 모든 레코드를 인덱스(=`trust_grade` 체크 없음). 단 이 파일의 계약은 "src를 그대로 materialize, row i==입력 line i". **여기서 B를 필터하면 row정렬 계약(#4가 의존)을 깬다.**
- 그러므로 B 제외는 **상류**에서: `chunks_with_vectors.jsonl`를 만드는 머지/임베딩 단계(P1)에서 B를 빼고 쓴다. 의료 경로(`build_medical_index`)는 이미 A만 임베딩. **전체 경로(embed_waves/embed_batch)가 B를 포함했는지 확인** 후, 포함됐으면 머지 시 제외.
- 파일: P1의 머지 글루(상류). `embed/faiss_index.py`는 건드리지 말 것.

## #6 [CONFIRMED] ad_review가 doc_type=law 고정
- 검증: `search/ad_review.py`에서 주 검색이 `flt={"doc_type":"law"}` k=4, 보충이 unscoped k=2뿐 → 표시기준·고지의무 등 **행정규칙 과소회수**.
- 수정: `flt={"doc_type":["law","admrule"]}`(retriever `build_filter`가 리스트 match-any 지원) 또는 law/admrule 2-pass 각 충분 k.
- 파일: `search/ad_review.py`.
- 참고: `AD_LAW_TITLES` 상수는 **정의돼 있으나 실제 스코프에 안 쓰임**(죽은 상수). 실제 의료 고정은 `doc_type=law` 필터 + 판정 프롬프트다. 거길 고칠 것.

## #7 [CONFIRMED 메커니즘 / 데이터 미확인] 판례 law_kind silent-miss
- 검증: `search/retriever.py`의 `ALLOWED_FILTER_KEYS`에 `law_kind`가 있고 **모든 doc_type에 일괄 적용**. 판례에도 그대로 걸림. "판례 law_kind={형사,일반행정}뿐"이라는 **데이터 분포는 코드만으론 미확인** → 코퍼스로 확인할 것. 메커니즘상 silent-miss 취약점은 실재.
- 수정: 판례는 별도 `case_category` 메타, 또는 판례에 law_kind 필터 비적용.
- 파일: `search/retriever.py`, (파서 메타).

## #8 [위치 수정] 메타 보일러플레이트 노이즈
- 검증: **retriever에 BM25가 없다**(dense 단독). 따라서 "BM25 색인 텍스트를 본문만으로" 수정할 **대상 코드가 현재 없음.**
- 재배치:
  - BM25 보일러플레이트 제외 = **P2(하이브리드 BM25 구축)에서** 색인 텍스트를 본문·조문제목·법령명만으로.
  - dense 벡터의 헤더 희석 A/B = **`embed/chunk.py`**(헤더 생성단), retriever 아님. 헤더 포함/제외 임베딩 비교는 임베딩 재생성이 필요하니 비용·범위 고려.
- 파일: (P2) BM25 색인부, `embed/chunk.py`(dense 헤더 정책).

## 완료 기준 (DoD)
