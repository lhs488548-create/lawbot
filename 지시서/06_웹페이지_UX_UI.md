# P6 — 웹페이지: 로그인 없이 바로 쓰는 깔끔한 테스트 페이지 (UX 최우선)

> 사용자 강조: **별도 로그인 없이 바로 사용·테스트**, **깔끔하게**, **사용자 입장에서 UX/UI 불편 없게.** 우선순위 높음.
> 본 지시서는 실제 `web/chat.html`·`api/main.py` 코드 대조 검증(2026-06-29)으로 정정됨.

## 목적
방문자가 키 입력·로그인 없이 즉시 질문하고 광고문안을 검토하는 군더더기 없는 단일 페이지.

## 현재 상태 (검증 결과 — 오해 정정)
- `web/chat.html`에 **탭 2개(법률 질문/광고문안 검토)·SSE 스트리밍·인용 배지(trust_grade A/B)는 실재.**
- ❗ **"diff뷰 이미 있음"은 사실이 아님** — chat.html에 diff/교정본 뷰 없음. 백엔드는 `corrected_copy`·`original_text`를 반환하나 페이지가 안 그림. → **신규 구현 대상.**
- ❗ **"raw JSON 노출"은 현재 chat.html엔 없음** — 답변은 텍스트+배지로 렌더. (옛 페이지 기억일 수 있음.) → **이 항목은 작업 불필요.**
- ❗ `/v1/ask`·`/v1/ad-review`는 **`require_key`로 401 강제** → 현재 페이지는 키 입력 필드가 있고 키 없으면 막힘. **"바로 사용"은 새 작업이 필요**(아래 무인증 데모).
- ❗ **`/` 루트 라우트 없음** — 테스트 페이지는 **`/chat`**. (`/`는 404)
- ❗ **chat.html 광고검토 렌더 실제 버그**: 카드가 `it.severity`·`it.note`·`it.text`를 읽는데 백엔드 `Issue`엔 그런 필드가 **없음**(실제 필드: `verdict`(위반/위반소지/주의/적정/확인필요)·`severity`(high/medium/low/none)·`rationale`·`law_basis`·`suggested_fix`). → 카드가 빈 본문/잘못된 색으로 나옴. **재바인딩 필요.**

## 대상 파일
- `web/chat.html` — 다듬기 대상.
- `web/console.html` — 키 관리(테스트 페이지와 분리, 노출 안 함).
- `api/main.py` — 무인증 데모 경로, `/chat` 라우트(이미 있음).
- `api/auth.py`(`issue_key`/`verify`), `api/db.py`(tier `free`=30/min) — 데모키 재사용.

## 작업 단계
1. **로그인 없는 데모 접근 (구체 방식 — 이걸로 확정)**:
   - 현재 `/v1/ask`·`/v1/ad-review`가 `require_key`라 익명 불가. **스키마 변경 없이** 다음으로 구현:
     - 서버 기동 시 환경변수(예: `LAWBOT_DEMO=1`)면 **free 티어 데모키를 1개 발급(또는 재사용)** — 기존 `auth.issue_key` 사용, `ApiKey`에 만료 컬럼 추가 불필요.
     - **무인증 라우트 `GET /v1/demo-token`** 신설 → `{key}` 반환(데모키만).
     - `chat.html`이 로드 시 키 필드가 비어 있으면 `/v1/demo-token`을 호출해 그 키를 Bearer로 사용. 키 입력 UI는 테스트 페이지에서 **숨김**.
   - 남용 방지: free 티어가 이미 키별 30/min 레이트리밋(`api/db.py`) → 그대로 적용. (전역/IP 추가 캡은 P7.)
   - 대안(택1, 비권장): `/v1/ask`·`/v1/ad-review` 의존성을 `optional_key`로 바꾸고 IP 버킷 캡 추가 — 라우트 보안 정책이 바뀌므로 위 데모키 방식을 우선.
2. **깔끔한 레이아웃**: 상단 제품명+한 줄 설명, 단일 컬럼, 모바일 대응, 탭 2개(기본=법률 질문), 예시 질문 칩 3~4개(클릭 자동입력).
3. **답변 UX**:
   - 스트리밍(SSE) 로딩 상태 표시(빈 화면 금지). (raw JSON 제거 작업은 불필요 — 이미 없음.)
   - 인용 배지: 법령·조문 + trust_grade + **P5의 신호등 색상·신뢰점수**(P5 완료 후), 클릭 시 source_url 펼침.
   - **주 근거 / 참고 인접법 분리 표시**(P3).
   - disclaimer + "AI 생성" 표기 하단(과하지 않게).
4. **광고검토 UX (버그 수정 포함)**:
   - 카드 필드를 **재바인딩**: 색/라벨은 `verdict`(위반/위반소지/주의/적정/확인필요), 본문은 `rationale`+`law_basis`+`suggested_fix`. `it.note`/`it.text`/Korean-`severity` 가정 제거.
   - **심각도 순 정렬**(`verdict` 또는 `severity`).
   - **교정본(corrected_copy) 렌더 + 복사/다운로드**, 가능하면 원문 대비 diff(신규).
5. **에러·빈응답 친절화**: 비법률 입력엔 안내+예시 재노출, 근거불충분/타임아웃/레이트리밋 초과는 사람이 읽는 메시지(스택트레이스 금지).
6. **반응성**: 로딩 스켈레톤, 모바일 정상, 대비·글자크기 충분.

## 완료 기준 (DoD)
- 키/로그인 없이 `/chat`에서 질문→답변, 광고문안→검토 끝까지 수행(데모키 자동).
- 광고검토 카드가 verdict 색·본문(rationale 등)·교정본을 올바로 표시(현 버그 해소).
- 인용 배지·신호등·주/참고 분리 표시.
- 모바일/데스크톱 정상, 로딩·에러 친절, 데모 레이트리밋 동작.

## 진행 상황 (2026-06-29)
- ✅ **무로그인 데모 API**: `config.DEMO_MODE`(env `LAWBOT_DEMO=1`) + `api/main.demo_or_require_key`(키 있으면 키 사용, 없고 데모면 익명 허용+IP 레이트리밋, 데모 off면 401). `/v1/ask`·`/v1/ad-review`에만 연결(나머지·프로덕션 키 유지, 기본 off=무회귀). 검증: import OK, 2곳만 적용.
- ✅ **사전 확인**: 답변 경로(ask)는 7GB서 OOM 없이 동작(parents 비호출 — P1 정정 참조). 즉 페이지에서 ask·ad-review 실제 테스트 가능.
- ◻️ **남은 P6**: chat.html — 데모 시 키입력 숨김+자동 무키 호출, **광고검토 카드 필드 재바인딩**(`verdict`/`rationale`/`law_basis`/`suggested_fix`; 기존 `note`/`severity` 오참조 버그), 교정본 렌더, 인용 신호등(P5 후). 경로 `/chat`(루트 `/` 없음). 그 후 `LAWBOT_DEMO=1 LAWBOT_INDEX=full LAWBOT_HYBRID=1`로 uvicorn 기동→실제 테스트.

## 환각 방지 체크
- 페이지는 **실제 API 응답 필드만** 렌더(응답 스키마를 코드에서 확인 후 바인딩) — 현재 광고검토 버그가 바로 이 위반 사례.
- 테스트 페이지 경로는 `/chat`(루트 `/` 없음). 데모 무인증 경로는 P7 레이트리밋과 연동.
