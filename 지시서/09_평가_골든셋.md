# P9 — 평가: 골든셋 · Recall@5/MRR@3 회귀 측정

## 목적
"검색이 정말 사용자가 원하는 답을 주는가"를 추측이 아니라 숫자로 본다. 모든 검색 개선(P2 하이브리드, 정규화, 라우팅, ANN)의 전/후를 같은 기준으로 비교한다.

## 현재 상태 (검증 결과)
- `eval/golden_set.jsonl` **존재하나 20행**(30~50 아님). `run_eval.py` docstring도 "20 lawyer-style questions".
- 실제 스키마(=`GoldenItem`): `{id, query, doc_type, expect_title_contains[], expect_article, expect_keywords[], as_of_date, notes}`. **`chunk_id` 필드 없음.** "분야 라벨"에 가장 가까운 건 `doc_type`(law/precedent/admrule/ordinance).
- `run_eval.py` **런타임 경로는 이미 FAISS**(`--mode direct`는 `search.statutes.statutes_search` 호출, `--mode http`는 `/v1/statutes/search`). 단 **docstring/주석이 Qdrant·gpt-4o로 stale**(코드는 정상).
- 지표는 **Hit@K / MRR@K / Article-hit@K**(기본 K=8). **Recall@5/MRR@3라는 이름의 출력은 없음.**

## 대상 파일
- `eval/golden_set.jsonl` — 20행 → 확장/검수.
- `eval/run_eval.py` — 런타임 FAISS 정상. **stale docstring/주석만 정정**(Qdrant·gpt-4o 언급 제거).

## 작업 단계
1. **골든셋 점검·확장**:
   - 현재 20행 → 범용에 맞게 **분야 다양화**(의료 외 노동/세무/형사/부동산) 30~50쌍 이상.
   - 각 항목은 **실제 스키마**로: `query`(구어), `doc_type`(=분야 라벨), `expect_article`·`expect_title_contains[]`·`expect_keywords[]`(정답 근거), `as_of_date`, `notes`. ⚠️ `chunk_id` 필드는 하네스가 안 읽으니 넣지 말 것.
   - **법 검수**가 이상적(틀린 정답 라벨은 평가를 오염).
   - **스냅샷 해시 핀 고정**: 현재 훅 없음(신규). 예) `config.FAISS_INDEX`의 sha256을 리포트 헤더/사이드카에 기록 — 위치를 구체화하거나 안 하면 생략.
2. **지표**:
   - 하네스가 내는 **Hit@K / MRR@K**를 쓴다. @5를 원하면 **`-k 5`** 플래그 필수(기본 8). "Recall@5/MRR@3"라는 이름을 원하면 코드 추가 필요.
   - LLM 비용 0의 retrieval-only: **`--mode direct`**(서버·키 불필요, 의료 FAISS 인덱스에 바로). `--mode http`(기본)는 서버+키 필요.
   - 게이트 발동률, top_score 분포(P4-#2 임계값 보정용).
3. **회귀 루프**: 변경 전/후 같은 골든셋으로 측정, 표로 비교. 개선/퇴행을 수치로 기록.

## 완료 기준 (DoD)
- 실행: `cd /home/user1/lawbot && .venv/bin/python -m eval.run_eval --mode direct -k 5` (서버 없이 의료 FAISS에 바로). Hit@5/MRR@5 출력.
- 하이브리드/정규화/라우팅 적용 전후 비교표 산출(같은 골든셋).
- run_eval.py의 stale Qdrant/gpt-4o docstring 정정.
- (선택) 스냅샷 해시 기록 — 훅 신설 시.

## 사용처
- P2: 하이브리드/정규화 효과 증명.
- P4-#2: top_score 분포로 게이트 임계값 결정.
- P1: ANN recall이 brute-force 대비 ≥0.98인지 확인.
- 제품: "lawbot.org보다 낫다"를 주장하려면 이 수치가 근거.

## 환각 방지 체크
- 골든셋 정답 라벨을 모델이 만들지 말 것(법 검수 또는 코퍼스 실제 조문으로). 틀린 라벨은 평가 전체를 거짓으로 만든다.
- run_eval 런타임은 **이미 FAISS** — "Qdrant 경로를 새로 만들" 필요 없음. docstring/주석만 정정.
- 지표 이름을 코드에서 확인(Hit@K/MRR@K) — Recall@5라는 출력은 없으니 `-k`로 K만 맞추거나 코드 추가.
