# P2 — 하이브리드 검색(BM25+RRF) · 질의 정규화 · 도메인 라우팅

## 목적
dense 단독 검색의 노이즈/누락을 잡는다. 법령명·조문번호 같은 정확 문자열 매칭을 BM25로 보강하고, 구어 질의를 법조문체로 정규화하며, 범용 코퍼스에서 분야를 좁히는 라우팅을 추가한다.

## 선행조건 (중요)
- **P4-#2(MIN_RETRIEVAL_SCORE 게이트)를 반드시 먼저 수정.** 안 고치면 RRF 점수(최대 ~0.016)가 게이트(0.20)에 막혀 모든 질의가 "근거불충분"이 된다.
- P1 전체 인덱스 권장(의료 인덱스로 먼저 프로토타입 가능).

## 대상 파일
- `search/retriever.py` — 검색 본체. 현재 dense-only, 사후 메타필터.
- `config.py` — 신규 플래그(하이브리드 on/off, RRF k, 부스트 가중치) 상수 추가.
- (라우팅) `search/` 내 신규 모듈 또는 retriever 내부 함수.

## 작업 단계
1. **BM25 인덱스 구축**:
   - 색인 텍스트는 **본문 + 조문제목 + 법령명만**(헤더 보일러플레이트 제외 — 버그 #8). "이 조문은 …를 규정한다" 같은 문장이 IDF를 망친다.
   - 한국어 토크나이저 함정 주의: SQLite FTS5 기본 unicode61은 형태소를 못 쪼갠다("의료광고를" ≠ "의료광고"). 선택:
     - (권장·ROI 높음) **법령명·조문번호 키워드 사전 + 정규식 부스트**("제56조", "의료법", 고시번호 패턴을 정확 매칭해 가산점).
     - 또는 rank-bm25(파이썬) 인메모리 BM25, 또는 FTS5 trigram(부분매칭, 랭킹 주의).
   - C 확장(mecab/khaiii)은 NCP 단일서버 빌드비용 커서 후순위.
2. **RRF 융합**: dense 순위와 BM25 순위를 Reciprocal Rank Fusion(k=60 기준)으로 합친다. 최종 점수는 **정렬용**으로만 쓰고, 게이트는 dense 원점수에 적용(버그 #2 정책).
3. **질의 정규화**: 구어→법 개념 재작성 또는 HyDE(가상 답 생성 후 임베딩). LLM 호출이 늘면 지연/비용 고려해 옵션화. 정규화 전/후 Recall 비교(P9).
4. **도메인 라우팅 (범용 핵심)**:
   - ⚠️ **현재 payload엔 "분야(의료/노동/세무…)" 키가 없다.** `ALLOWED_FILTER_KEYS = {doc_type, jurisdiction, law_kind, effective_from, parent_id}`(retriever.py)뿐. 따라서 분야로 바로 못 거른다. 둘 중 하나:
     - (a) **빌드 시 payload에 분야 필드 신설**(예: domain/subject) 후 그걸로 스코프, 또는
     - (b) 질의를 분야 분류 → 기존 `doc_type`/`law_kind`로 매핑(완전치 않음).
   - 무필터 136만 전수 검색은 노이즈 폭발 → 라우팅으로 후보 공간 축소.
   - 라우팅 실패(분야 불명)는 전체 검색으로 폴백.
5. retriever `search()` **시그니처와 Hit 구조는 유지**(계약). 내부만 교체.

## 완료 기준 (DoD)
- 하이브리드 on/off를 config로 토글 가능.
- "의료법 제56조 안 잡힘" 같은 정확매칭 케이스가 top-5에 들어옴(P9 골든셋으로 확인).
- 게이트가 RRF 점수를 막지 않음(전 질의 차단 없음).
- `python -m search.retriever --selftest` 회귀 없음.
- P9 기준 하이브리드 적용 전후 Recall@5/MRR@3 향상 수치 첨부.

## 구현 완료 (2026-06-29) — 하이브리드 1차
- ✅ BM25: `embed/build_bm25.py`(SQLite FTS5 **unicode61** + 검색 시 **prefix `term*`**로 한국어 어미/조사 매칭) → `artifacts/full_index/bm25.sqlite`(rows 1,238,122, FAISS 행과 정렬). trigram은 2글자 한국어 매칭 불가라 폐기.
- ✅ retriever: `_bm25_search`(불용어 필터+구두점 strip, ttl 가중 5.0), `_is_citation_like`(제N조·○○법 감지), `_hybrid_search`(가중 RRF, dense 1.0/BM25 `RRF_W_BM25`). **타겟형**: 인용/법명 질의에만 BM25 융합(순수 구어체는 dense 단독 — BM25 판례노이즈 차단). hit.score는 실제 코사인(reconstruct)이라 게이트(#2)와 호환.
- ✅ **측정(31문항 골든셋)**: dense 61.3% → **타겟 하이브리드 64.5%**(Hit@5), MRR 0.467→0.516, Article-hit 33%→67%, **격식 질의 회귀 0**. 동일가중 RRF는 48%로 악화(노이즈) → 타겟+가중이 정답. 최적 `RRF_W_BM25=2.0` 기본값.
- ⚠️ **남은 약점(다음 레버 = 질의 정규화)**: 구어-어휘 격차(월급→임금, 잘림→해고, 성인→성년)는 BM25·dense 둘 다 못 잡음 → 동의어/구어→법조문체 정규화 필요. 통치조문 vs 판례 우선순위(cite-001 근로기준법 연차)도 정규화/부스트 추가 영역. 도메인 라우팅은 미구현(범용 단일 인덱스에선 후순위).

## 환각 방지 체크
- BM25/RRF 추가가 retriever 공개 시그니처를 바꾸지 않는지 확인(상위 모듈 무영향).
- 토크나이저 선택의 실제 한국어 매칭 동작을 작은 샘플로 검증 후 채택(추정 금지).
- 질의 정규화 LLM 출력도 근거검증 흐름을 깨지 않도록.
