# P1 — 전체 FAISS 인덱스 빌드 + 무결성 게이트 + ANN

## 목적
완료된 136만 임베딩(`artifacts/embeddings.jsonl`)을 텍스트(`artifacts/chunks.jsonl`)와 합쳐 **전체 코퍼스 FAISS 인덱스**를 만들고, retriever가 이 전체 인덱스를 쓰도록 연결한다. 규모가 크므로 brute-force 대신 ANN 적용을 결정·구현한다.

## 선행조건
- `artifacts/embeddings.jsonl` = 1,357,018줄(검증 완료), `artifacts/chunks.jsonl` 동일 chunk_id 세트.
- 빌드 계약: `docs/_FAISS_BUILD_CONTRACT.md` 준수(정규 산출물 `chunks_with_vectors.jsonl`, 512d, L2정규화).

## 대상 파일 (현재 코드 확인 후 작업)
- `embed/faiss_index.py` — `build_index`/`load_index`/`export_sqlite_vec`. 빌드 로직.
- `embed/build_medical_index.py` — 머지 글루 패턴 참고(이미 chunks⨝vectors 패턴 있음).
- `embed/chunk.py` — 청크 텍스트/메타 소스.
- `search/retriever.py` — 인덱스 로드 경로.
- `config.py` — `FAISS_INDEX`, `FAISS_META`, `CHUNKS_VEC_JSONL`, `EMBED_DIM`(512), `PARENTS_JSONL`.

## 작업 단계
1. **머지 글루 작성** (`embed/build_full_index.py` 신규 또는 build_medical_index 패턴 재사용):
   - `embeddings.jsonl`({chunk_id, vector})와 `chunks.jsonl`(텍스트+payload)을 **chunk_id로 조인** → 정규 산출물 `chunks_with_vectors.jsonl`({chunk_id, doc_id, parent_id, text, payload, vector}). 계약의 레코드 형태 그대로.
   - 스트리밍 처리(136만, 메모리 안전). chunk_id 불일치/누락은 카운트·로그.
   - **B등급 제외**: `payload.trust_grade == "B"`(본문 없는 별표/라벨)는 **이 머지 단계(상류)에서** 제외하고 쓴다. ⚠️ `faiss_index.py`에는 넣지 말 것(그 파일은 "src 그대로 materialize, row정렬 보존" 계약이라 거기서 필터하면 #4 검증을 깬다). 전체 임베딩 경로(embed_waves/embed_batch)가 B를 포함했는지 먼저 확인 후, 포함됐으면 머지에서 제외.
2. **무결성 게이트** (검증 결과: 일부 이미 구현됨):
   - ✅ **이미 있음**(재작업 금지): `index.ntotal == len(metas)`(build·load 모두), chunk_id 중복 거부, 차원 검증, row 정렬 — `embed/faiss_index.py`에 구현됨.
   - ➕ **추가 필요**: add 직후 **벡터 L2노름 ≈ 1(±1e-3) 사후검증**만 신설(현재 build는 정규화는 하나 사후 assert는 없음).
   - 파일 쓰기는 이미 atomic.
3. **인덱스 타입 결정·구현**:
   - 기본은 계약대로 `IndexFlatIP(512)`지만 136만은 쿼리당 수백 ms~수 초. **ANN 적용**:
     - 1차 권장: `IndexHNSWFlat`(M=32, efConstruction=200, efSearch 64~128) 또는 `IndexIVFFlat`(nlist≈√N, nprobe 튜닝).
     - recall은 brute-force 대비 ≥0.98 목표(P9 골든셋으로 측정).
   - **메모리 확인**: Flat 약 2.8GB + 메타. NCP RAM 확정 전이면 일단 Flat로 빌드해 동작 확인 후 ANN로 교체. (RAM 결정은 P8 참조.)
4. **retriever 연결**: `config.FAISS_INDEX`가 전체 인덱스를 가리키게 하고 `search/retriever.py`가 이를 로드. 의료 인덱스는 별도 경로로 보존(롤백용).
   - ⚠️ **서빙 RAM 경고(실측)**: 전체 인덱스 빌드 완료 = `artifacts/full_index/`(index.faiss **2.54GB** + meta.jsonl **3.17GB**, ntotal **1,238,122** = 136만 − B등급 118,896). 현재 `load_index`는 **meta를 통째로 RAM에 적재**(→ ~6GB) + 인덱스 2.5GB = **~8.5GB 상주** → **7GB WSL 박스에선 OOM.** 서빙하려면 (a) 서버 RAM ≥16GB(P8), (b) **meta를 디스크 오프셋 조회로 전환(저RAM retriever)**, (c) 양자화 중 택1. 빌드 자체는 메모리 안전한 `embed/build_full_index.py`로 완료됨(스트리밍).

## 완료 기준 (DoD)
- `chunks_with_vectors.jsonl` 생성, 라인수 = 임베딩(− B등급 제외분).
- `python -m embed.faiss_index --selftest` 통과(빌드 무결성).
- L2노름 사후검증 통과 출력 첨부.
- ⚠️ **`search.retriever --selftest`로는 전체 인덱스를 검증할 수 없음**(이 셀프테스트는 인메모리 더미 4-doc 인덱스 + 임베딩 스텁이라 실인덱스를 안 읽어 항상 통과 = 가짜 green). 대신 **실인덱스 라이브 질의**로 확인: `python -m search.retriever "의료광고 심의 기준"` → 전체 코퍼스(법령/판례 등)가 실제로 나오는지 확인.

## 검증 명령
```
.venv/bin/python -m embed.faiss_index --selftest        # 빌드 무결성 (OpenAI 호출 없음)
.venv/bin/python -m search.retriever "의료광고 심의 기준"   # 실인덱스 라이브 (질의당 임베딩 1회 과금)
```

## 환각 방지 체크
- build_index의 실제 시그니처·산출 경로를 코드에서 확인 후 글루를 맞출 것.
- ANN로 바꾸려면 **`build_index` 본체를 수정**해야 함(현재 `IndexFlatIP` 하드코딩, 인덱스 타입 파라미터 없음). 읽기측(`load_index`/retriever)은 `faiss.read_index`라 타입 무관 → 안 건드려도 됨. score 의미(내적=코사인) 유지 확인(HNSW/IVF도 IP metric).
- 비용: **빌드 자체는 0**(벡터 이미 있음). 단 실인덱스 라이브 검증 질의는 질의당 임베딩 1회 과금.
- `embeddings.jsonl` 총 1,357,018줄은 이번 세션 무결성 검증에서 확인됨(라인=고유 chunk_id=코퍼스). 대용량이라 재집계 시 시간 소요.
