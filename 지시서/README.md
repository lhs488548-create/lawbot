# lawbot 작업지시서 — 범용 법률 검색·자문 제품 만들기

이 폴더는 lawbot을 **의료특화 → 범용 한국법 검색·자문 제품**으로 완성하기 위한 파트별 작업지시서다. 각 파일은 한 엔지니어(또는 한 에이전트)가 독립적으로 집어 실행할 수 있도록 자기완결적으로 썼다.

배경·근거의 전체 그림은 `docs/lawbot_최종_정리본.md`와 `docs/lawbot_완성도_개선_정리.md`에 있다. 이 지시서는 "무엇을, 어떤 파일에서, 어떻게, 무엇을 통과하면 끝"만 다룬다.

---

## 0. 절대 원칙 (모든 파트 공통 — 위반 시 작업 무효)

### 0-1. 환각 금지
- **코드를 고치기 전에 반드시 해당 파일을 직접 읽고 현재 상태를 확인한다.** 이 지시서에 적힌 라인 번호·함수명은 작성 시점 기준 참고값이다. **실제 라인은 현재 코드에서 다시 확인**하고, 다르면 현재 코드를 따른다.
- 존재하지 않는 함수·상수·엔드포인트를 가정하지 않는다. `config.py`의 상수는 거기서 import만 하고 재정의하지 않는다.
- "아마 이럴 것"으로 진행 금지. 모르면 grep/read로 확인한다.
- 외부 사실(법령 내용, 시행일, 조문 번호)을 모델 지식으로 만들어내지 않는다. 반드시 코퍼스/law.go.kr에서 가져온다.

### 0-2. 근거 없으면 침묵 (제품 핵심 가치)
- LLM이 답을 만드는 모든 경로는 **Structured Outputs(strict JSON) + 인용 사후검증(retrieved id와 불일치 인용 폐기)**을 유지한다. 이미 `search/rag.py`·`search/ad_review.py`에 패턴이 있으니 그대로 따른다.
- 검색 근거가 부족하면 지어내지 말고 "확인 필요"로 처리한다.

### 0-3. 비용 가드
- OpenAI 유료 호출은 **견적 → 승인 → 실행** 순서. 테스트는 의료 인덱스나 소량으로. 임베딩 전량 재실행 금지(이미 완료됨, `artifacts/embeddings.jsonl` 1,357,018개).
- 테스트에서 실제 OpenAI 호출은 셀프테스트당 1~2회 이내.

### 0-4. 완료(DoD) 보고 규칙
- "완료"는 **각 파트의 검증 명령을 실제로 실행하고 그 출력을 첨부**해야 인정된다. 실행 안 한 추정 완료 금지.
- 기존 셀프테스트(`python -m search.retriever --selftest`, `python -m embed.faiss_index --selftest` 등)에 회귀가 없어야 한다.

### 0-5. 환경
- WSL `/home/user1/lawbot`. Windows 도구는 UNC `\\wsl.localhost\Ubuntu\home\user1\lawbot\...`, 실행은 `.venv/bin/python`.
- 한글 경로/인자는 명령줄에 직접 넣지 말고 `config` 상수나 `.py` 파일 경유.
- **선행 점검**: 모든 검증 명령이 `.venv/bin/python`을 쓴다. 시작 전 `.venv/bin/python -V`로 venv 무결성 확인(간혹 WSL I/O 오류로 깨질 수 있음).
- **현황(검증 2026-06-29)**: 라이브 FAISS 인덱스는 **의료 14,070개뿐**(`artifacts/의료관련/faiss/`). 전체 136만 인덱스는 **아직 없음**(P1에서 빌드). 따라서 "전체 코퍼스"를 전제로 한 DoD는 P1 완료 후에만 충족. `embeddings.jsonl`(13.5GB)·`chunks.jsonl`(3.4GB)은 chunk_id 스키마 일치 확인됨(머지 전제 OK).

## 2. 제품 방향 (확정)
- **범용 한국법 검색·자문 제품.** 의료특화 아님. lawbot.org 기능 대부분 보유, 부족분(하이브리드·신뢰점수UI) 추가, 코퍼스 규모(136만)로 차별화.
- **lawbot = 순수 법률 엔진 API.** UI/세션/파일/PDF재생성은 MediLaw(Node) 또는 별도. 단 이번 단계의 테스트 페이지는 lawbot이 직접 제공(로그인 없이 바로 테스트).
- 비교·성능 목표는 골든셋으로 측정해 증명(P9).

## 3. 페이지 정책 (사용자 강조)
- **별도 로그인 페이지 없이 바로 사용·테스트 가능**하게. 데모는 키 없이(서버측 데모키 또는 IP 레이트리밋) 동작.
- **UX/UI가 사용자 입장에서 불편하지 않게**가 최우선. 자세한 건 P6.
