# 02. 데이터 완벽 분석 — 6개 ZIP 법률 코퍼스

> **문서 목적**: `D:\법파일\*.zip` 에 담긴 6개 공공저작물 법률 코퍼스(판례·자치법규·행정규칙)를 전수 분석하여, RAG(Retrieval-Augmented Generation) 기반 한국 법률 봇 구축에 필요한 전체 규모·구조·품질·통합 데이터모델·청크 스키마·전처리 파이프라인·임베딩 비용을 한 문서로 정리한다.
> **근거 데이터**: 정확 통계는 `D:/법파일/_samples/corpus_stats.txt` 를 직접 인용한다. 코퍼스 구조·필드는 멀티에이전트 분석(phase2)을, 설계 시사점은 phase3(임베딩·벡터DB·청킹·평가)을 반영한다.
> **라이선스**: 전 코퍼스 본문은 저작권법 §7(보호받지 못하는 저작물: 법령·자치법규·고시·훈령·판결 등)에 해당해 저작권이 미발생(자유 재배포 가능). ※용어상 "공공저작물 자유이용"은 §24조의2이나 결론은 동일. 출처 — 판례·행정규칙 = `law.go.kr`(국가법령정보센터), 자치법규 = `elis.go.kr`(ELIS). 원천 프로젝트 = `github.com/legalize-kr`.

## 0. 요약 (Executive Summary)

| 항목 | 수치 |
|---|---|
| ZIP 파일 수 | 6개 |
| 디스크 총량 | 약 9.3 GB |

| **RAG 인덱싱 대상 청크(현행 조문 + 판례)** | **약 341만** (3,287,831 조문 + 123,743 판례) |
| 판례 | 123,743건 |
| 자치법규 현행 조문(4개 시도 합계) | 1,977,948 |
| 행정규칙 현행 조문 | 1,309,883 |

**핵심 결론 4가지**

1. **데이터는 이미 RAG에 최적화되어 있다.** 자치법규·행정규칙은 `current.articles/*.txt` 로 조문 단위 사전 분할이 되어 있어 별도 청킹 로직이 거의 불필요하다. 청크 경계 = 법적 인용 단위(조문)와 일치한다.
2. **"항목수 폭증" 함정에 주의해야 한다.** ZIP 총 항목수(약 1,394만)의 대부분은 과거 버전 스냅샷(`versions/`)이다. 인덱싱 대상은 `current.articles`(현행)뿐이며, 이를 필터링하지 않으면 비용·노이즈가 4~8배 폭증한다.
3. **데이터 커버리지 한계를 명확히 인지해야 한다.** 본 코퍼스에는 **국가 1차 법령(법률·시행령·시행규칙)이 없고**, 자치법규는 **17개 시도 중 4곳(광주·세종·전남·전북)뿐**이다. "한국 법률 봇"이 아니라 "**판례 + 4개 시도 자치법규 + 국가 행정규칙 검색·인용 봇**"으로 포지셔닝해야 정직하다.
4. **품질 이슈는 정형적이고 처리 가능하다.** 판례 섹션 결측·비식별화, 자치법규 항/호 단위 과편화, 행정규칙 조문 과편화(95.8%)·고시 본문 결측(27%) 등은 모두 전처리 파이프라인으로 흡수 가능하다.

**경로 규칙 (4단계 분류)**
```

  ├─ current.txt                                   # 현행 전체 본문

```

**조문 txt 포맷**
```
# {법령명}
```

## 3. 강점·품질 이슈

### 3.1 코퍼스 공통 강점

- [x] **조문 단위 사전 분할** — 자치법규·행정규칙은 `current.articles/*.txt`가 이미 조(條)/항 단위로 쪼개져 청크 경계가 법적 인용 단위와 일치. 별도 청킹 비용 ≈ 0.
- [x] **자기완결 헤더** — 각 조문 txt 1·2행에 법령명+조문번호+제목이 포함되어 청크 단독으로도 출처 맥락 유지(임베딩 컨텍스트 손실 적음).
- [x] **풍부한 정형 메타데이터** — `meta.json`(지역·종류·시행일·개정이력·trust_grade·첨부)으로 하이브리드 검색의 패싯 필터를 무료 확보.
- [x] **판례 디렉토리 분류** — 사건종류·법원등급이 디렉토리 구조 자체로 분류되어 별도 분류 모델 불필요.
- [x] **판례 섹션 헤더 명시** — `## 판시사항/판결요지/...` 정규식만으로 섹션 청킹 가능. 판시사항·판결요지는 사람이 쓴 고품질 짧은 요약.
- [x] **안정적 식별자** — `alr_bdt_id`/`precSeq`가 원문 역추적·중복판정 키로 사용 가능. **git 커밋 해시는 force-push로 불안정**하므로 인용 식별은 반드시 ID 기반으로.
- [x] **단일·명확한 라이선스** — 전량 공공저작물(저작권법 §7)이라 재배포·임베딩·파인튜닝 법적 리스크 낮음.

| 필수 | **과거버전 인플레이션** (자치법규 항목수 5~7배) | 무분별 인제스트 시 중복·구버전 검색 오염 | `current.articles`만 인덱싱, 과거버전은 메타·temporal 인덱스로만 |
| 필수 | **행정규칙 조문 과편화 95.8%** (동일 조문번호가 항/호 단위 다수 파일로 분산; 47개 txt ↔ 17개 distinct 조문) | 동일 조문이 마이크로 청크로 흩어져 검색 정밀도·리콜 저하, 벡터 수 부풀림 | `(alr_bdt_id + 조문번호)` 기준 **재병합** 전처리 |
| 권장 | **행정규칙 전체의 27%(5,676건) 본문 결측**(대부분 고시 — 결측의 약 87%) | 고시·공고는 별표 중심이라 조문 파서가 본문 추출 실패 → 검색 누락 | 최소 메타(규칙명+부처+종류+시행일) 제목 임베딩으로 색인, 본문은 source_url 폴백 |
| 권장 | **판례 섹션 광범위 결측** (판시사항 38%·판결요지 44%·참조판례 65% 누락, 특히 하급심) | "요지만 검색" 전략이 절반에서 깨짐 | 결측 시 판례내용에서 의사-요지 추출, 섹션 보유 플래그 메타화 |
| 권장 | **비식별화(○○○) 35%** | 임베딩 노이즈, 인물 추적 불가, 동명 충돌 | 일관 토큰 `[PERSON_n]` 치환, 인물 질의 가드레일 명시 |
| 권장 | **참조조문/참조판례 평문** ('형법 제306조', '대법원 80도1448 판결') | ID 링크 없어 그래프·인용검증 불가 | 정규식 + 법령명 사전으로 정규화·엔티티링킹 |
| 권장 | **별표·별지 본문 결측** (라벨만 존재) | 요율·금액·기준표 답변 불가 | 라벨 키워드 색인 + "별표 본문 미수록" 가드, ELIS 링크 폴백 |
| 주의 | **trust_grade 변별력 0** (세종 전량 A, 행정규칙 전량 B) | 코퍼스 간 의미 불일치, 신뢰도 필터 무력 | `source_corpus` 별도 필드 사용, trust_grade는 코퍼스 내 비교에만 |
| 주의 | **ID 체계 이질성** (자치법규=숫자, 행정규칙=`ADMRULE:n`, 판례=precSeq) | 통합 doc_id 충돌 위험 | 네임스페이스 접두 통합 규칙 |
| 주의 | **region 의미 충돌** (행정규칙 sido=부처) | 통합 시 필드 의미 혼선 | doc_type별 jurisdiction 매핑 분기 |
| 주의 | **인라인 `<개정 …>` 주석 + 원문자 ①②** | 임베딩 노이즈, 날짜 포맷 불일치(`2025.04.01` vs `2022.1.3`) | 정규식 추출 → `amendment_dates` 메타화, 본문 정규화 |
| 주의 | **데이터 신선도** (`imported_at` 1회성 스냅샷) | 법령 수시 개정 → stale 위험 | 정기 증분 재인제스트(법령ID/날짜 델타) |

```jsonc
  "issuer": "법원명 또는 소관부서",
```

```jsonc

  "article_title": "조문제목 | null",
  "text": "임베딩 대상 본문 (헤더·개정주석 분리 후)",

  // payload = 필터 가능 메타 (벡터DB)
  "payload": {
    "doc_type": "...",
    "source_corpus": "...",
    "sido": "...", "sigungu": "...", "competent_ministry": "...",
    "kind": "...", "court_level": "...",
    "effective_from": "YYYY-MM-DD",   // 시점 필터
    "promulgated_on": "YYYY-MM-DD",
    "trust_grade": "A | B",
    "source_url": "...",
    "title": "...",
    "has_section": { "판시사항": true, "판결요지": false }  // 판례 결측 플래그
  }

```

### 4.3 인용(Citation) 포맷 규칙

| 유형 | 포맷 |
|---|---|
| 법령(조문) | `{ord_name} {조문번호}({조문제목}) [시행 {effective_from}] (출처: {source_url})` |
| 판례 | `{법원명} {선고일자} 선고 {사건번호} 판결 [{사건명}] (출처: law.go.kr precSeq={precSeq})` |

> 응답에 `doc_id`·`chunk_id`를 함께 반환해 역추적을 보장. 인용 식별은 **항상 안정 키(alr_bdt_id/precSeq/사건번호)**를 사용(git 해시 금지).

## 5. 청크 스키마 & 청킹 전략

### 5.1 유형별 청킹 규칙

| 유형 | 기본 청크 단위 | 추가 처리 |
|---|---|---|
| **자치법규** | `current.articles/*.txt` 1파일 = 1청크 (조/항) | 첫 2행 헤더 → 메타 승격, 본문만 임베딩. 항 단위 과편화 시 조 단위 재조립. 빈 본문 파일 필터 |
| **행정규칙** | `(alr_bdt_id+조문번호)`로 **재병합**한 조문 = 1청크 | 95.8% 과편화 재조립이 핵심 전처리. 평균조문 62.2는 파편 오집계(실제는 더 적음) |
| **판례** | `## 섹션` 단위 1차 분할 | 판시사항/판결요지 = 통째 1청크(고가중치). 판례내용 = 800~1,200토큰·200 오버랩 2차 분할(부모-자식). DRM 완화 위해 사건번호+사건명 포함 ~150자 문서요약 주입(SAC) |

### 5.2 부모-자식(Small-to-Big) & SAC

- **부모-자식**: 자치법규는 조문=부모, 항·호=자식. 자식으로 정밀 검색하고 부모(조문 전체)를 LLM에 전달해 컨텍스트 보존.
- **SAC(Summary-Augmented Chunking)**: 법률 문서의 보일러플레이트 유사성이 Document-Level Retrieval Mismatch(DRM)를 유발(NDA류에서 95%+). 각 청크 앞에 ~150자 문서요약 프리펜딩 시 DRM 약 절반 감소(arXiv 2510.06999). 자치법규는 `ord_name`+조문제목으로, 판례는 사건번호+사건명으로 구현.

## 6. 전처리 파이프라인

```
[1] ZIP 스트리밍 추출
      └─ 전남(3.5GB)·전북(2.8GB) 대용량은 메모리 스트리밍 필수
      └─ current.articles 외 versions/·README.md·미분류 항목 필터
            (자치법규 5~7배 인플레이션 제거 → 인덱싱 대상 1/5~1/8)
            ↓
[2] 유형별 파서
      ├─ 판례: YAML 프론트매터 + ## 섹션 분할
      └─ 법령: 헤더 2줄 + meta.json 조인
            ↓
[3] 텍스트 정규화
      ├─ 원문자 ① ② → (1) (2)
      ├─ <개정 YYYY.M.D.> 주석 추출 → amendment_dates 메타화, 본문에서 제거
      ├─ 비식별화 ○○○ → [PERSON_n] 일관 치환
      └─ 공백·유니코드 정규화
            ↓
[4] 통합 doc_id / chunk_id 부여 + 공통 스키마 매핑
      └─ region → jurisdiction (doc_type별 분기)
            ↓
[5] 청크 조립
      ├─ 행정규칙: (alr_bdt_id+조문번호) 과편화 조문 재병합
      └─ 판례: 섹션 2차 청킹 + SAC 요약 주입
            ↓
[6] 중복 제거 (해시 + precSeq/alr_bdt_id, 지역 접두어 정규화 near-dup 클러스터링)
            ↓
[7] 임베딩 + payload 적재 (벡터DB)
            ↓
[8] 별표·histories·과거버전 → 메타/temporal 인덱스로만 보존
```

**전처리 체크리스트**

- [ ] `current.articles` 외 모든 디렉토리(versions, README) 인덱싱 제외 확인
- [ ] 행정규칙 조문 과편화 재병합 로직 검증(distinct 조문번호 ↔ 파일 수 비율 측정)
- [ ] 행정규칙 27%(대부분 고시) 무조문 법령 메타 색인 폴백 적용
- [ ] `<개정>` 주석·원문자·비식별화 정규화
- [ ] 참조조문/참조판례 평문 → ID 엔티티링킹
- [ ] effective_from 누락·estrev_label null·sigungu null 빈값 처리
- [ ] `competent_tel` 등 PII 노출 정책 검토
- [ ] 지역 접두어 정규화 후 near-duplicate 클러스터링(광주∩세종 동일제목 221건 확인됨)

## 8. RAG 설계 시사점 (phase3 종합)

### 8.1 검색 아키텍처
- **하이브리드(BM25+dense) → RRF 융합 → 크로스인코더 리랭킹** 2단계가 한국어 법률 검색에 우수(Recall@5 하이브리드 리랭킹이 단일 방식 능가).
- 메타데이터 **사전 필터(pre-filter)** 필수: `sido/sigungu`·`ord_kind`·`effective_from`·`court_level`·`사건종류`를 payload 인덱싱 → "광주 조례만", "대법원 세무 2010년 이후" 같은 질의 사전 좁히기.
- **현행본(current_version)만 기본 검색**, 시점 질의는 `effective_from`+`histories[]` 기반 temporal 보조 인덱스.

### 8.2 환각·인용 보장
- 상용 RAG 법률 도구조차 환각 17~33%(Stanford JELS 2025) → **RAG=환각 해결**로 전제 금지.
- **인용 강제**: JSON Schema 구조화 출력 + 사후 인용검증(생성된 조문/사건번호가 코퍼스에 실재하는지 `alr_bdt_id`/`precSeq` 대조). correctness뿐 아니라 faithfulness(모델이 실제 근거에 의존했는지)도 평가.
- 평가 기준: 한국어는 **KBL**(11k 조문(articles) + 15만 판례 RAG 세팅), 검색은 **LegalBench-RAG** 방식 Recall@k/Precision@k, 통합 ablation은 **LRAGE**.

### 8.3 법적·윤리 가드레일
- **변호사법 §109** — "법률 조언"이 아닌 "법령·판례 정보 제공/인용"으로 포지셔닝. 구체 사안 결론·문서작성은 회피.
- **인공지능기본법(2026.1.22 시행)** — 생성형 AI 사실 사전 고지 + 결과물 AI 생성 표시(위반 시 과태료 3천만원 이하).
- **면책 고지 자동 삽입** — 모든 답변에 "법률자문 아님 + 출처·시행일 명시 + 변호사 상담 권고".
- **출처 표기** — 공공저작물(저작권법 §7) + 원천(law.go.kr/ELIS/legalize-kr), MIT(구조/메타) 이원 라이선스 승계 고지.
