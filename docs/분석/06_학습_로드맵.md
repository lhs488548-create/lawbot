# 06. 완전 초보자용 학습 로드맵 — 나만의 lawbot API를 이해하고 만들기까지

> **누구를 위한 글인가**: 프로그래밍·AI를 **하나도 모르는 일반인**이, 이 프로젝트(한국 법률 RAG API)를 **스스로 이해하고 만들 수 있을 때까지** 무엇을 어떤 순서로 배워야 하는지 정리한 지도.
> **읽는 법**: 위에서 아래로 **순서대로**. 각 단계는 ①왜 배우나(이 봇의 어디에 쓰이나) ②무엇을(쉬운 비유 → 정확한 정의) ③어디서(무료·한국어 우선) ④스스로 확인 질문 으로 구성.
> **중요**: 모든 걸 "마스터"할 필요 없습니다. 각 단계의 **굵게 표시한 핵심**만 이해하면 다음으로 넘어가도 됩니다. 막히면 그 부분만 다시, 더 쉽게 물어보세요.
> **목표 스택(검증 완료)**: Python · 벡터DB(Qdrant 또는 pgvector) · 임베딩(KURE-v1) · 리랭커(bge-reranker-v2-m3-ko) · LLM API · FastAPI · Docker · 클라우드.

## 0. 큰 그림 먼저 — "lawbot이 답하는 과정" 한 장으로

배우기 전에, 우리가 만들 것이 **무슨 일을 하는지** 먼저 그림으로 잡으면 모든 공부가 "아, 이거 때문에 배우는구나"로 연결됩니다.

```
사용자: "전입신고는 언제까지 해야 하나요?"
   │
   ▼
[1] 질문을 '숫자 목록'으로 변환 (임베딩)          ← 5,6단계
   │
   ▼
[2] 340만 개 법조문·판례 중 '의미가 비슷한' 것 검색 (벡터DB)  ← 4,7단계
   │
   ▼
[3] 후보를 다시 정밀 정렬 (리랭커)                ← 7단계
   │
   ▼
[4] 뽑힌 근거 조문 + 질문을 LLM에 줘서 '근거 기반 답변' 생성  ← 6,7단계
   │
   ▼
[5] 답변 + 출처(법령·판례 링크) + 면책고지를 인터넷으로 반환 (API)  ← 8,9단계
```

> 이 7개 상자가 곧 학습 단계 ④~⑨입니다. 그 앞 ①~③은 "이 상자들을 이해하기 위한 기초"입니다.

학습 단계 전체 지도:

| 단계 | 주제 | 수준 | 이 봇에서의 역할 |
|---|---|---|---|
| ① | 컴퓨터·인터넷 원리 | 초등 | 모든 것의 바탕 |
| ② | 기초 수학·논리 | 초·중 | 임베딩·검색의 직관 |
| ③ | 프로그래밍 첫걸음(파이썬) | 중 | 모든 코드의 언어 |
| ④ | 데이터·데이터베이스 | 중·고 | 법조문을 담고 찾기 |
| ⑤ | AI·머신러닝 기초 | 고 | "학습"이 뭔지 |
| ⑥ | NLP·LLM·임베딩 | 고·대 | 말을 숫자로, 답을 생성 |
| ⑦ | RAG (이 봇의 심장) | 대 | 검색+생성 결합 |
| ⑧ | 웹·API·백엔드(FastAPI) | 대 | 인터넷에 서비스화 |
| ⑨ | 배포·운영(Docker·클라우드) | 대·실전 | 24시간 가동 |
| ⑩ | 종합 — 만드는 순서표 | 실전 | 전부 연결 |

**예상 기간(하루 1~2시간 기준)**: ①~③ 약 4~6주, ④~⑤ 약 4주, ⑥~⑦ 약 6~8주, ⑧~⑨ 약 4~6주. 총 **약 5~7개월**이면 "이해하고 직접 만든다" 수준에 도달합니다. (빨리 가려면 ③⑥⑦⑧에 집중)

## ① 컴퓨터와 인터넷은 어떻게 작동하나 (초등 수준)

**①-1. 왜 배우나**
우리 봇은 "프로그램"이고 "인터넷"으로 서비스됩니다. 컴퓨터가 정보를 어떻게 저장·처리하고, 인터넷에서 어떻게 주고받는지를 모르면 8·9단계(API·배포)가 외계어가 됩니다.

**①-2. 무엇을**
- **컴퓨터의 4요소(쉬운 비유)**: CPU=요리사(계산), RAM=도마(작업 중 잠깐 올려두는 곳, 끄면 사라짐), 저장장치(SSD)=냉장고(꺼도 남음), 입출력=손과 눈.
  - 정확히: 프로그램은 저장장치에 파일로 있다가, 실행하면 RAM에 올라와 CPU가 처리한다.
- **파일과 폴더**: 우리 데이터(`D:\법파일`의 zip, `.md`, `.txt`)가 전부 파일이다. 경로(`D:\법파일\분석\01_...md`)는 파일의 "주소".
- **이진수(0과 1)**: 컴퓨터는 모든 걸 0/1로 저장한다. 글자도 숫자로 바뀐다(예: 'A'=65). → 6단계 "글자를 숫자로"의 뿌리.
- **인터넷의 기본**: 인터넷은 컴퓨터끼리 편지를 주고받는 우체국망.
  - **클라이언트/서버**: 손님(브라우저·앱)=클라이언트, 음식점(우리 봇)=서버. 손님이 "요청(request)" 보내면 서버가 "응답(response)" 준다. ← API의 핵심.
  - **IP 주소·도메인**: IP=건물 주소 숫자, 도메인(`naver.com`)=그 주소의 별명.
  - **HTTP/HTTPS**: 편지 양식(규칙). HTTPS=자물쇠 채운 안전 편지.

**①-3. 어디서 (무료)**
- 생활코딩 "프로그래밍이란?" / "WEB1 - 인터넷" (opentutorials.org) — 한국어, 가장 쉬움
- 유튜브 "10분 테코톡 - HTTP", "컴퓨터 구조 기초"
- 책: 『모두의 컴퓨터 과학(CS)』 또는 『그림으로 배우는 Http & Network Basic』

**①-4. 스스로 확인**
- RAM과 SSD의 차이를 냉장고 비유로 설명할 수 있다?
- "클라이언트가 서버에 요청을 보내고 응답을 받는다"를 음식점으로 설명할 수 있다?
- HTTPS의 's'가 무엇을 뜻하는지 안다?

## ② 기초 수학·논리 — 겁먹지 말 것 (초·중 수준 + 직관)

**②-1. 왜 배우나**
"임베딩"과 "벡터 검색"(이 봇의 심장)은 결국 **숫자 목록(벡터) 사이의 거리·각도**를 재는 일이다. 고등 수학을 풀 필요는 없고, **직관**만 잡으면 된다.

**②-2. 무엇을 (핵심만)**
- **집합과 분류**: 우리 데이터는 판례/자치법규/행정규칙으로, 또 지역·종류(조례·규칙)로 나뉜다. "필터링"=조건에 맞는 부분집합 고르기.
- **좌표와 벡터(가장 중요한 직관)**:
  - 초등 비유: 지도에서 한 장소 = (가로 3, 세로 5) 두 숫자. 이 숫자 쌍이 **2차원 벡터**.
  - 두 장소가 "가깝다"=좌표 거리가 작다. → 검색은 "질문 좌표와 가까운 법조문 좌표 찾기".
  - 정확히: KURE-v1 임베딩은 한 문장을 **1024개 숫자(1024차원 벡터)**로 바꾼다. 사람은 1024차원을 못 그리지만, "가까우면 의미가 비슷하다"는 직관은 2차원과 똑같다.
- **거리와 유사도**: 유클리드 거리(자로 잰 직선거리), **코사인 유사도**(두 화살표가 이루는 각도가 작을수록 1에 가깝다=비슷). 벡터 검색은 주로 코사인을 쓴다.
- **확률의 직관**: LLM은 "다음에 올 단어"를 **확률**로 고른다. "오늘 날씨가 ___"에서 '좋다'가 80%, '슬프다'가 1% 같은 식. → 6단계 LLM 이해의 바탕.
- **로그·지수는 '그렇구나' 정도만**: 큰 수를 다루는 도구. 깊게 안 가도 됨.

**②-3. 어디서**
- 칸아카데미 한국어 — 벡터, 좌표평면, 확률 기초
- 유튜브 3Blue1Brown "선형대수의 본질"(자막) — 벡터를 **그림으로** 이해(강력 추천, 직관만)
- 책: 『프로그래머를 위한 선형대수』는 나중에, 지금은 영상으로 충분

**②-4. 스스로 확인**
- "문장을 1024개 숫자로 바꾼다"가 무슨 뜻인지 한 줄로 말할 수 있다?
- 코사인 유사도가 1에 가깝다 = 두 문장이 어떻다는 뜻?
- LLM이 다음 단어를 "확률로 고른다"가 무슨 느낌인지 안다?

## ④ 데이터와 데이터베이스 — 법조문을 담고 찾기 (중·고 수준)

**④-1. 왜 배우나**
우리는 **57,350개 법령·12.4만 판례·약 340만 조문**을 다룬다. 이걸 "어떻게 저장하고 빠르게 찾는가"가 봇 성능의 절반이다. 특히 **벡터DB**는 이 봇의 핵심 부품.

**④-2. 무엇을**
- **데이터 형식**:
  - **텍스트/파일**: 지금 우리 데이터(`.txt` 조문, `.md` 판례).
  - **JSON**: 키-값 구조(`meta.json`). 파이썬 딕셔너리와 똑같이 생김 → API가 주고받는 표준 형식.
- **데이터베이스(DB)란**: 데이터를 체계적으로 저장·검색하는 창고 + 사서.
  - **관계형DB(SQL, 예: PostgreSQL)**: 엑셀 표처럼 행·열. "광주 + 조례 + 2023년 이후"같은 **정확한 조건 검색**에 강함. 우리 봇의 메타데이터(지역·종류·시행일) 필터에 사용.
  - **SQL 기본**: `SELECT * FROM 법령 WHERE 지역='광주' AND 종류='조례'` — 영어 문장처럼 읽힘.
- **벡터DB(이 봇의 심장 부품)**:
  - 일반DB는 "단어가 똑같은 것"을 찾지만, 벡터DB는 **"의미가 비슷한 것"**을 찾는다.
  - 비유: 도서관에서 "전입신고"라는 글자가 든 책이 아니라, **내용이 비슷한 책**을 꺼내주는 마법 사서.
  - 검증된 후보: **Qdrant**(검색 전용, 빠르고 메타 필터 강함) 또는 **pgvector**(PostgreSQL 확장 — SQL 필터 + 벡터를 한 곳에서). 입문엔 둘 다 좋고, "정확한 필터 + 의미 검색"을 한 DB에서 하고 싶으면 pgvector가 단순.
  - **하이브리드 검색**: 키워드 검색(BM25, 조문번호·정확한 단어) + 벡터 검색(의미)을 합치면 법률에 특히 강하다. → 7단계에서 다룸.

**④-3. 어디서**
- 생활코딩 MySQL/관계형 데이터베이스, "SQL 기초" (한국어)
- Qdrant 공식 튜토리얼(quickstart) / pgvector GitHub README
- 유튜브 "벡터 데이터베이스가 뭐길래"

**④-4. 스스로 확인**
- JSON과 파이썬 딕셔너리가 왜 닮았는지 안다?
- 일반DB(똑같은 단어)와 벡터DB(비슷한 의미)의 차이를 도서관 비유로 설명?
- "광주 조례만, 2023년 이후"는 SQL 필터 / "전입신고와 비슷한 조문"은 벡터 검색 — 이 분담이 이해된다?

## ⑤ 인공지능·머신러닝 기초 — "학습"이 뭔가 (고 수준)

**⑤-1. 왜 배우나**
임베딩 모델·LLM·리랭커가 전부 "**머신러닝으로 학습된 모델**"이다. "모델이 학습한다"의 의미를 알아야 6단계가 술술 풀린다.

**⑤-2. 무엇을 (개념만, 수식 X)**
- **AI ⊃ 머신러닝 ⊃ 딥러닝 ⊃ LLM** 의 포함관계.
- **머신러닝이란**: 규칙을 사람이 일일이 안 짜고, **데이터에서 패턴을 스스로 찾게** 하는 것.
  - 비유: 강아지/고양이 사진 수만 장을 보여주며 "맞다/틀리다"를 반복하면, 스스로 구별 규칙을 익힌다.
- **모델·파라미터·학습**: 모델=거대한 함수, 파라미터=그 함수의 수많은 손잡이(가중치), 학습=정답과 비교해 손잡이를 조금씩 돌려 오차를 줄이는 과정.
- **딥러닝·신경망**: 사람 뇌 뉴런을 흉내 낸 층(layer)을 여러 개 쌓은 모델. "딥"=층이 깊다.
- **학습 vs 추론**: 학습=손잡이 맞추는 훈련(비쌈, 우리는 안 함). **추론=다 배운 모델을 가져다 쓰기**(우리가 할 일). → 우리는 **이미 학습된 KURE-v1·LLM을 가져다 추론만** 한다(그래서 GPU·비용 부담이 작다).
- **파인튜닝**: 이미 학습된 모델을 우리 분야(법률)에 맞게 살짝 더 훈련. (선택 사항, 나중에)
- **과적합·일반화, 학습데이터/평가데이터**: "외워버리기 vs 진짜 이해". 평가의 필요성. → 7단계 평가로 연결.

## ⑥ 자연어처리·LLM·임베딩 — 말을 숫자로, 답을 생성 (고·대 수준)

**⑥-1. 왜 배우나**
이 봇의 두 뇌가 여기 있다: ① **임베딩**(질문·법조문을 숫자로 바꿔 검색 가능하게) ② **LLM**(찾은 근거로 답 생성). 2단계 직관 + 5단계 학습개념이 여기서 합쳐진다.

**⑥-2. 무엇을**
- **자연어처리(NLP)**: 컴퓨터가 사람 말을 다루는 분야. 토큰화(문장을 토큰 단위로 쪼갬), 형태소 분석(한국어는 조사·어미가 붙어서 **nori/mecab/kiwi** 같은 한국어 형태소 분석기가 BM25 검색에 중요).
- **임베딩(Embedding) — 핵심**:
  - 정의: 단어·문장을 **의미를 담은 숫자 벡터**로 바꾸는 것. 비슷한 의미는 비슷한 벡터.
  - 우리 모델 **KURE-v1**: 고려대 NLP&AI Lab이 bge-m3를 한국어로 파인튜닝, **1024차원**, 최대 **8192토큰**, MIT 라이선스, MTEB-ko(한국어 검색 벤치) 1위. ※"법률 전용 SOTA"는 아니고 **범용 한국어 검색 최상급**(법률은 평가 도메인 중 하나) — 출시 전 법률 데이터로 직접 성능 측정 권장.
  - 왜 중요: 우리 약 340만 조문·판례를 전부 임베딩해 벡터DB에 넣어두면, 질문도 임베딩해서 "가까운 것"을 즉시 찾는다.
- **LLM(거대 언어 모델)**:
  - 정의: 방대한 텍스트로 학습해 **다음 단어를 확률로 이어 쓰는** 초대형 신경망(GPT·Claude·Gemini, 또는 한국어 특화 HyperCLOVA X·EEVE·Qwen 등).
  - **토큰·컨텍스트 창**: LLM은 한 번에 정해진 양(토큰)만 본다. 그래서 "관련 법조문만 골라" 넣어줘야 한다(=RAG가 필요한 이유).
  - **프롬프트**: LLM에게 주는 지시문. "아래 조문만 근거로, 모르면 모른다고 답하고, 출처를 표기하라" 같은 설계가 환각을 줄인다.
  - **환각(hallucination)**: 그럴듯한 거짓말. 법률에선 치명적 → 7단계에서 인용·검증으로 억제.
- **API로 LLM 쓰기**: 직접 LLM을 돌리지 않고, OpenAI/Anthropic 등의 API로 호출(가장 쉬움). 비용·속도·한국어 성능을 보고 선택.

**⑥-3. 어디서**
- 유튜브 "트랜스포머/어텐션 쉽게", 허깅페이스(HuggingFace) 한국어 입문 글

## ⑦ RAG — 이 봇의 심장 (대학 수준)

**⑦-1. 왜 배우나**
**이 프로젝트 그 자체**가 RAG다. 4·6단계(벡터DB·임베딩·LLM)를 **하나의 파이프라인**으로 엮는 법을 여기서 배운다.

**⑦-2. 무엇을**
- **RAG(검색증강생성) 정의**: LLM이 "기억"으로 답하지 않고, **외부 지식(우리 법조문 DB)을 먼저 검색**해 그 근거로 답하게 하는 구조. → 환각↓, 출처↑, 법령 개정 즉시 반영.
- **인덱싱 파이프라인(준비 단계, 1회성)**:
  1. **로드**: zip에서 조문·판례 추출(과거버전 제외, current만).
  2. **청킹**: 자치법규·행정규칙은 **이미 조문 단위로 분할**돼 있어 그대로 청크. 판례는 섹션(판시사항/판결요지/…)으로 나눔. (2단계 문서 §5 참조)
  3. **임베딩**: 각 청크를 KURE-v1으로 벡터화(약 340만 개).
  4. **적재**: 벡터 + 메타데이터(지역·종류·시행일·출처)를 벡터DB(Qdrant/pgvector)에 저장.
- **질의 파이프라인(서비스 단계, 매 질문)**:
  1. **질문 임베딩** → 2. **벡터 검색**(top-50 후보) + **BM25 키워드 검색** 병합(하이브리드) → 3. **리랭커**(bge-reranker-v2-m3-ko)로 top-8 정밀 정렬 → 4. **프롬프트 조립**(질문 + top-8 근거) → 5. **LLM 생성**(근거 기반 답 + 출처 표기) → 6. **사후 인용 검증**(답에 인용된 조문/사건번호가 실제 DB에 있는지 확인).
- **리랭킹**: 1차 검색은 빠르지만 거칠다. 리랭커가 후보를 다시 읽고 정밀 순위를 매겨 정확도를 올림.
- **메타데이터 필터**: "광주 조례만", "2023년 이후 시행"처럼 검색 전 범위를 좁힘(4단계 SQL/페이로드 필터).
- **평가(매우 중요)**: 만들고 끝이 아니라 **얼마나 맞는지** 측정.
  - 검색 품질: Recall@k, nDCG. 답변 품질: 충실성(faithfulness, 근거에 기반했나)·정답성.
  - 도구: RAGAS, LegalBench-RAG 방식. **골든셋**(정답이 있는 질문 100~300개)을 직접 만들어 회귀 테스트.
- **법률 특화 가드레일**: 면책고지 자동 삽입, "법률 자문 아님" 표기, 모법(국가 법령) 미수록 시 경고(데이터 갭, 05 문서).

**⑦-3. 어디서**
- LangChain / LlamaIndex 공식 "RAG Quickstart"(둘 중 하나로 시작)
- 인프런·패스트캠퍼스 "RAG 챗봇 만들기" 한국어 강의
- 논문 맛보기: LegalBench-RAG(arXiv 2408.10343), KBL(2410.08731) — 어렵지만 "법률 RAG는 이렇게 평가하는구나" 감만
- 실습: 우리 데이터 1,000개만 떼어 미니 RAG를 굴려보기(전체 340만 전에 소규모로)

**⑦-4. 스스로 확인**

## ⑧ 웹·API·백엔드 — 인터넷에 서비스화 (대학 수준)

**⑧-1. 왜 배우나**
RAG 파이프라인을 내 PC에서만 돌리면 나만 쓴다. **API**로 감싸야 앱·웹·남이 인터넷으로 호출할 수 있다. 이게 "배포 가능한 lawbot **API**"의 'API' 부분.

**⑧-2. 무엇을**
- **API란**: 식당의 "메뉴판+주문구". 손님이 정해진 양식으로 주문(요청)하면 정해진 양식으로 음식(응답)을 준다. 우리 봇은 `POST /ask {"question": "..."}` → `{"answer":"...", "sources":[...]}`.
- **HTTP 메서드·상태코드**: GET(조회)·POST(보내기), 200(성공)·404(없음)·500(서버오류).
- **REST API / JSON**: 가장 흔한 API 양식. 입출력은 JSON(4단계).
- **FastAPI(검증된 선택)**: 파이썬으로 API를 빠르고 쉽게 만드는 도구. 자동 문서(Swagger), 입력 검증(Pydantic), 비동기 지원.
  - 핵심 구성: **엔드포인트**(`/ask`, `/health`), **요청 모델**(질문 형식 검증), **스트리밍**(답을 토큰 단위로 흘려보내 빠르게 보이게, SSE), **인증**(API 키), **레이트리밋**(과다호출 방지), **CORS**(웹에서 호출 허용).
- **비동기(async)**: 한 손님 음식 기다리는 동안 다른 주문도 받기 → 동시 사용자 처리.
- **테스트**: `pytest`로 "이 질문엔 이 출처가 나와야 한다" 자동 검사.

**⑧-3. 어디서**
- FastAPI 공식 튜토리얼(한국어 번역 있음, 매우 친절) — 必
- 점프 투 FastAPI(wikidocs) / 노마드코더·인프런 FastAPI 강의
- 실습: `/health`(살아있나)와 `/ask`(7단계 RAG 호출) 두 개부터

- `POST /ask`에 무엇을 보내고 무엇을 받는지 JSON으로 그릴 수 있다?
- 200과 500 상태코드의 차이?
- 스트리밍이 사용자 경험을 왜 좋게 하는지?

## ⑨ 배포·운영 — 24시간 인터넷에 띄우기 (대학·실전 수준)

**⑨-1. 왜 배우나**
내 PC를 끄면 봇도 죽는다. **클라우드 서버**에 올려 누구나 24시간 접속하게 하는 것이 "**배포**". 04 문서의 배포 계획을 실제로 실행하는 단계.

**⑨-2. 무엇을**
- **Docker(컨테이너)**: 내 프로그램 + 필요한 모든 것(파이썬·라이브러리)을 **상자 하나로 포장**. "내 PC에선 됐는데…" 문제를 없앤다. 어디서든 똑같이 실행.
  - `Dockerfile`(포장 방법서), 이미지(포장된 상자), 컨테이너(실행 중인 상자), `docker compose`(여러 상자—API+벡터DB—를 한 번에).

  - 입문·저비용: **Railway·Render·Fly.io**(깃 연결하면 자동 배포). 벡터DB는 매니지드(Qdrant Cloud) 또는 같은 서버.

  - 확장: AWS/GCP + 컨테이너 오케스트레이션(나중에).
- **도메인·HTTPS**: 도메인(`mylawbot.kr`) 구입 → 서버에 연결 → **HTTPS 인증서**(Let's Encrypt 무료)로 자물쇠.
- **환경변수·비밀관리**: API 키 등은 코드에 박지 말고 환경변수로(절대 GitHub에 올리지 말 것).
- **관측성·운영**: 로그(무슨 일이 있었나), 모니터링(살아있나·느린가), 에러 추적, 비용 모니터링, 백업.

- **보안·법규(05·01 문서)**: 변호사법 면책고지, AI기본법 고지·표시, 개인정보, 레이트리밋·남용 방지.

## ⑩ 종합 — 이 lawbot을 만드는 실제 순서표 (실전)

이제 배운 것을 **만드는 순서**로 다시 엮습니다. 이게 곧 04 문서의 "0→1 액션 플랜"과 연결됩니다.

### 10-1. 추천 학습→제작 통합 로드맵

| 주차(대략) | 학습 | 동시에 만드는 것(미니 실습) |
|---|---|---|
| 1~4주 | ①②③ 컴퓨터·수학직관·파이썬 | 파이썬으로 corpus_stats·meta.json 읽기 |
| 5~7주 | ④ 데이터·DB·벡터DB | 조문 1,000개를 SQLite/JSON으로 다뤄보기 |
| 8~11주 | ⑤⑥ ML개념·임베딩·LLM | KURE-v1로 문장 100개 임베딩 + LLM API "Hello" |
| 12~17주 | ⑦ RAG | **데이터 1,000개로 미니 RAG** 완성(검색→리랭크→생성) |
| 18~21주 | ⑧ FastAPI | 미니 RAG를 `/ask` API로 감싸기 |
| 22~26주 | ⑨ Docker·배포 | Docker 포장 → Railway 무료 배포 → 전체 340만으로 확장 |

> 핵심 전략: **처음부터 340만 개를 다 넣지 말 것.** 1,000개 → 1만 개 → 전체 순으로 키우면 실수 비용이 작다.

### 10-2. 실제 구축 단계(요약, 03·04 문서와 연결)

1. 데이터 부족분 먼저 보강(05 문서: 국가 법령 본문·전국 자치법규) — **만들기 전에 수집**
