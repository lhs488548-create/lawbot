# 09. 청킹·임베딩·헤더 설계 (lawbot.org 정렬 · 확정 spec)

> 작성 2026-06-15 · 근거: 웹 리서치(OpenAI embeddings, Anthropic Contextual Retrieval, legal RAG chunking) + lawbot.org 실측 + 원천데이터.
> 이 문서가 **청킹/임베딩/헤더의 단일 진실(SoT)**. 빌드 모듈(chunk/embed/header)은 전부 여기에 맞춘다.
> 비용 제약: OpenAI 크레딧 $20 → **3-small + 결정적 헤더 + Batch**로 전량 ~$9. (재임베딩 금지 = 지금 확정 후 1회.)

## A. lawbot.org 정렬 — 우리가 따라갈 API 형식

lawbot.org는 챗봇이 아니라 **"AI가 가져다 쓰는 법률 데이터 인프라"**. 그 형식을 그대로 채택(+변호사용 부가기능).

| lawbot.org | 우리 엔드포인트 | 설명 |
|---|---|---|
| `POST /v1/statutes/search` | 동일 | 통합 법령+판례 검색(조/항/호·섹션 정밀) |
| `POST /v1/verify` | 동일 | **Citation Firewall** — AI 인용의 진위·적용·시점 검증 |
| `POST /v1/source-pack` | 동일 | 관련 법령·시행령·판례·행정규칙을 **LLM 인용가능 markdown 번들**로 |
| OpenAI호환 `/v1/embeddings` | 동일(내부 OpenAI 래핑) | 외부도 임베딩 호출 가능 |
| `as_of_date` 파라미터 | 모든 검색/검증에 | 특정 시점 현행법 조회 |
| 응답 메타 `trust_grade·source_url·license` | 모든 응답 | 감사가능성 |
| 콘솔(API Keys·usage·sources·sync logs) | `/console` + `/v1/keys` | 멀티테넌트 셀프서비스 |
| — (부가) | `POST /v1/ask` | RAG 질의응답(변호사 전문가모드) |
| — (부가) | `POST /v1/ad-review` | PDF 광고문구 법규검토+교정 |

**모든 응답 공통 메타**: `{trust_grade: "A|B", source_url, license, as_of_date, effective_from}`.

## B. 청킹 설계

### B-1. 단위 (parent-child 2계층)
법률문서는 교차참조 네트워크 → child로 정밀검색, parent로 생성/번들 맥락.

| 문서 | **child**(검색·임베딩 단위) | **parent**(생성·source-pack 단위) |
|---|---|---|
| 법령/자치법규/행정규칙 | **조문 1개** | 그 조문이 속한 **법령 전체**(또는 장) |
| 판례 | **섹션 1개**(판결요지/판례내용 등) | **판례 전체** |

- 원천이 이미 조문/섹션 단위 → 일반 스플리터 불필요. child_id ↔ parent_id 링크를 payload에 보관.
- 검색은 child 임베딩으로, 답변·번들은 parent 원문으로(작은 청크 정밀 + 큰 맥락).

### B-2. 2차 분할 (8,191 토큰 한도)
- 조문/섹션이 8,191토큰 초과(주로 긴 판례 섹션)만 분할: **창 1,000토큰 / 오버랩 150~200**, 가능하면 **문장 경계**(한국어는 `kss` 보조)로 자른다. 같은 부모로 묶고 `part_idx` 부여.
- 대부분 조문은 평균 100~130토큰 → 분할 불필요.

### B-3. 별표·별지 / 표
- 본문 있으면 **별도 child** 로 `kind="별표"` 태그 + 부모 법령 링크.
- 라벨/링크만 있으면 본문 임베딩하지 말고 **메타(파일링크)** 로만 보관, `trust_grade="B"`.

### B-4. 정규화(임베딩 전)
- `①②③`→`(1)(2)(3)` 병기, `<개정 …>` 개정주석은 **본문에서 분리해 메타**로(검색노이즈↓), 판례 `○○○`→`[당사자]`, 유니코드 NFC, 중복공백 정리.
- 원문은 보존(인용·source-pack용). 임베딩 텍스트는 정규화본.

## C. 임베딩 설계 (OpenAI)

| 항목 | 결정 | 근거 |
|---|---|---|
| 모델 | **text-embedding-3-small** | 1536d, $0.02/1M, 법률 정밀 충분·예산내 |
| 차원 | **1536 유지**(Matryoshka 축소 안 함) | 법률은 정밀 우선, 저장비용 문제없음 |
| 정규화 | OpenAI가 **L2=1로 정규화** | 코사인=내적 → Qdrant Cosine |
| 입력한도 | 8,191토큰 | 초과는 B-2로 분할 |
| 대량 | **Batch API**(50%↓, 24h) | 전량 ~$9 |
| 소량/증분 | sync(배치 2048) | 데모·갱신 |
| 멱등성 | **content hash 캐시** — 같은 텍스트 재임베딩 금지 | 비용·재실행 안전 |
| 컬렉션 | 한 컬렉션엔 한 모델만 | 모델 바꾸면 전량 재구축 필요 |

> ⚠️ 모델/차원/헤더 전략을 바꾸면 **전량 재임베딩(=비용 재발생)**. 그래서 이 문서로 **착수 전 1회 확정**.

## D. 헤더 거버넌스 ★(사용자 핵심 요구: "헤더 계속 관리·감독")

검색 품질의 절반은 "청크가 자기 맥락을 갖고 있는가"에서 갈린다(Anthropic **Contextual Retrieval**: 청크에 맥락 헤더를 붙이면 top-20 검색실패 **−35%**, 리랭크 결합 시 **−67%**). 우리는 이를 **비용 0의 결정적(규칙기반) 헤더**로 구현한다.

### D-1. 임베딩 텍스트 구조 (2층 헤더 + 본문)
```
<L1 인용헤더>   [법령] 도로교통법 > 제2장 > 제17조(자동차등의 속도) · 시행 2026-04-02 · 소관 경찰청
<L2 맥락헤더>   이 조문은 '도로교통법'(법률, 운전자 속도·통행 규율)의 일부로, 속도 제한과 위반효과를 규정한다.
<본문>          (정규화된 조문 본문)
```
- 판례 L1: `[판례] 대법원 2022-05-25 2022누50008 · 선거·특별 · 판결요지`
- **L2 맥락헤더는 LLM이 아니라 템플릿/규칙으로 생성** → 비용 0. (부모 제목 + 위치 + 핵심 엔티티(소관부처/지자체/사건종류) 조합.)

### D-2. 구조화 메타(payload) — 임베딩 텍스트와 별개로 항상 저장
`doc_id, parent_id, doc_type, title, jurisdiction, law_kind, chapter/section_path, article_no, article_title, effective_from, as_of, trust_grade, source_url, license, kind(본문|별표), part_idx`

### D-3. "관리·감독" = 강제 스키마 + 검증기 + 테스트 (CI·인제스트마다)
- `header/schema.py`: 헤더 필수필드·형식을 **단일 정의**(모든 doc_type 공통 규약).
- `header/validate.py`: 모든 청크가 (a) L1 필수필드(법령명/식별자·조문번호·시행일/선고일) 보유 (b) L2 맥락헤더 존재 (c) payload 필터키 완비 — 를 검사. **결측/형식오류 건수 리포트**.
- 인제스트 파이프라인과 `pytest`에서 **항상** 실행 → 헤더 깨지면 빌드 실패(=지속 감독).
- 청크 생성기는 헤더를 **한 곳(`header/build.py`)에서만** 만든다(중복·표류 방지).

## E. 검색 / as_of_date / 인용검증

### E-1. 검색 파이프라인
메타 사전필터(doc_type·jurisdiction·law_kind, **as_of_date ≤ effective 범위**) → dense(3-small) top-K → (옵션 Phase) BM25 하이브리드+RRF → (옵션) 리랭커 → **parent 승격**(child 히트→parent 원문 동봉) → 답변/번들.

### E-2. Citation Firewall (`/v1/verify`)
입력 인용{법령명·조문 또는 사건번호} → ① DB 존재 ② **law.go.kr OpenAPI(LAW_OC=<LAW_OC_REDACTED>, 검증됨)로 현행·문구 대조** ③ as_of_date 시점 유효성 → `{verified, trust_grade, current, source_url, note}`. 폐지·오인용·허위사건 탐지(콘솔 sync log에 기록).

### E-3. Source Pack (`/v1/source-pack`)
질의/사안 → 관련 child 검색 → parent 모아 **LLM이 그대로 인용가능한 markdown 번들**(법령>조문 원문 + 시행일 + 출처URL + trust_grade).

### E-4. 실시간 질의 서빙 (저지연 — "거의 즉시") ★
목표: 사용자 질문 → **첫 응답 토큰 1초 이내**. 코퍼스는 빌드 때 1회 색인, 런타임은 질문 1건만 처리.
1. **웜 싱글톤 + async**: OpenAI·Qdrant 클라이언트를 FastAPI `lifespan`에서 1회 초기화(요청마다 생성 금지), 엔드포인트 `async def`.
2. **질의 임베딩 캐시**: 질문 텍스트 해시 → 임베딩 LRU(+선택 Redis). 동일/반복 질문은 임베딩 재호출 0.
3. **Qdrant HNSW + payload 인덱스**: 필터(관할·doc_type·as_of_date) + dense 검색 수십 ms. top-K 6~8로 타이트.
4. **SSE 스트리밍**: `/v1/ask`는 gpt-4o-mini 답을 토큰 단위 스트리밍(`text/event-stream`) → 즉시 표시.
5. (선택) **답변 시맨틱 캐시**(Redis): 동일 질문 반복 시 LLM 생략.
- 지연 예산: 임베딩~100ms + 검색~30ms + LLM 첫토큰~300~500ms → 체감 <1초.
- `/v1/statutes/search`(검색형)는 LLM 없이 ~150ms. `/v1/ask`(답변형)만 LLM 스트리밍.
- 구현 위치: `search/retriever.py`(웜 클라이언트·캐시), `search/rag.py`(스트리밍 제너레이터), `api/main.py`(async·lifespan·SSE).

## G. 비용 가드 (요약)
- 전량 임베딩 전 **토큰 추정→예상비용 출력→사람 승인** 게이트.
- 데모/표본 검증은 ≤5만 청크(~$0.5). **전량(약 319만 청크 / 14.6억 토큰) = Batch 1회 약 $14.6**(sync $29.2 금지). 헤더는 결정적($0). large·LLM맥락 금지(예산). 표본 품질검증 후 전량 1회.
- content-hash 캐시로 재실행 시 재과금 0.

**한 줄 요약**: 조문/섹션=child·법령/판례=parent, 3-small(1536, 정규화)로 1회 Batch 임베딩, **2층 결정적 헤더(인용+맥락)를 스키마·검증기·테스트로 지속 감독**, lawbot.org 형식(statutes/search·verify·source-pack·embeddings·as_of_date·console)에 정렬.
