# lawbot 최종 정리본 — 성능 좋은 배포·API 가능한 lawbot을 만들기 위해 무엇이 더 필요한가

작성일 2026-06-29. 이 문서는 현재 lawbot 디렉토리의 모든 설계·전략 문서(`docs/*.md`), 다른 작업 세션에서 정리한 것들, lawbot.org 레퍼런스 조사, 그리고 코드베이스 실제 상태를 한데 모아 "지금부터 무엇을 더 해야 진짜 서비스가 되는가"를 정리한 것이다. 기존 문서들과 중복되는 세부는 출처만 적고, 판단과 다음 행동에 집중했다.

## 0. 한눈에 — 지금 위치

- **임베딩(재료) 100% 완료.** `artifacts/embeddings.jsonl` = 1,357,018개, 512차원, 무결성 검증 PASS(라인=고유 chunk_id=코퍼스, 중복 0·손상 0·차원오류 0·누락 0). 조례(자치법규)는 의도적으로 제외.
- **코드·API·데모·MCP·Docker 다 있음.** `/v1/ask`(SSE), `/v1/ad-review`, `/v1/verify`, `/v1/source-pack`, `/v1/statutes/search`, API키 인증, MCP 4도구, Caddy+FastAPI compose까지 구현되어 있고 281개 테스트 통과.
- **그러나 현재 라이브 인덱스는 "의료 14,070개"짜리뿐.** 방금 끝낸 136만 임베딩은 **아직 FAISS 인덱스로 안 묶였다.** retriever는 의료 인덱스만 로드 중.
- 그래서 **"임베딩 끝 = lawbot 완성"이 아니다.** 재료는 됐고, 그 위에 ① 전체 인덱스 빌드 ② 검색 알고리즘 ③ 제품화/운영 세 층이 남았다. 셋 다 기존 코드 위에 쌓는 것이지 새로 짓는 게 아니다.

핵심 한 줄: **지금 필요한 건 "더 많은 데이터"가 아니라 "전체 인덱스화 + 검색 품질 + 운영 안정성"이다.**

## 1. 사용자 질문에 대한 직접 답 — "FAISS 빌드 + 하이브리드 검색까지 했다면, 그 다음 뭐가 더 필요한가"

FAISS 인덱스 빌드와 하이브리드(BM25+RRF) 검색을 끝냈다고 가정해도, 진짜 배포 가능한 서비스가 되려면 아래가 더 필요하다. 중요도 순.

### 1-1. ANN 인덱스 — 136만 규모에선 brute-force가 느려서 필수
현재 설계는 `faiss.IndexFlatIP(512)` **brute-force**다. 의료 1.4만 개에선 쿼리당 5~50ms로 충분하지만, **136만 개에선 O(n)이라 쿼리당 500~5000ms**로 늘어난다(코드 조사 추정치). 동시 사용자가 붙으면 그대로 타임아웃 위험.
- 선택지: **FAISS IVF**(클러스터링, ~10배 빠름, recall 1~2% 손실) 또는 **HNSW**(지연 최저지만 메모리 더 큼). 설계 문서의 벤치마크 인용: 1000만 벡터에서 brute-force 12.4ms vs IVF 0.34ms vs HNSW 0.17ms. Flat은 실무적으로 ~50만~100만이 한계.
- 메모리: 136만 × 512 × 4바이트 ≈ **2.8GB**(인덱스만) + 메타 ~0.3GB ≈ 프로세스 ~3.1GB 상주. **NCP 서버 RAM을 먼저 확정**해야 인덱스 타입을 정할 수 있다(아래 7장).
- 즉 "하이브리드까지 했다"의 다음 1순위는 **인덱스 타입을 brute-force→ANN로 바꾸고 recall/지연을 측정**하는 것.

### 1-2. 리랭킹(rerank) — 조건부로
하이브리드로 후보를 잘 뽑아도, 상위 정렬 정확도를 끌어올리려면 cross-encoder 리랭커가 효과적이다(설계 문서: MRR@3 +17.2%p, Recall@5 +12.1%p). 단 NCP 단일서버엔 GPU가 없어 top-50 리랭크가 쿼리당 수백 ms. **무조건이 아니라 "결론이 특정 조문 1개로 갈리는 고위험 질의 + 1순위 점수가 애매할 때만" 조건부로** 켜는 게 맞다.

### 1-3. 골든셋 + 회귀 평가 — 이게 없으면 "좋아졌다"를 증명 못 한다
"구어 질의 → 정답 조문" 30~50쌍을 법으로 검수해 라벨링하고, **코퍼스 스냅샷 해시로 버전을 핀 고정**해 둬야 한다. 모든 검색 개선(하이브리드/리랭크/정규화)의 전후를 Recall@5·MRR@3로 정량 비교하는 기준선. **이것 없이는 검색 튜닝이 감으로 흘러간다.** 지금 의료 인덱스로 당장 만들 수 있다(임베딩 완주를 안 기다려도 됨).

### 1-4. 8대 핵심 버그 수정 — 정식화 전 필수 (특히 법적 리스크)
문서 `lawbot_완성도_개선_정리.md`에서 확인된 것들. 하이브리드를 켜기 전에 일부는 반드시 먼저 고쳐야 한다.

| # | 버그 | 영향 | 고칠 곳 |
|---|---|---|---|

### 1-5. 운영 인프라 — 서비스 등급이 되려면
- **런타임 비용 가드**: 현재 빌드 단계 가드만 있고 **API 키별 호출 쿼터·일일 토큰 상한이 없다.** ad_review 1건이 (주장 수 × (임베딩+검색+law.go.kr 최대 12회))를 태우므로 한 키가 폭주 가능. 키별 쿼터 + law.go.kr 호출에 타임아웃/캐시/서킷브레이커 필요.
- **메트릭/관측성**: 게이트 발동률, 인용 폐기율, as_of_date 미적용 비율, 임베딩 캐시 히트율, 쿼리 지연 히스토그램 — 현재 stdout 로깅뿐.
- **개정 동기화**: "시행일이 도래하면 미래 조문을 자동으로 현행으로 승격"하는 메커니즘 부재. law.go.kr OC는 있으나 갱신 신호·주기 명세가 없음.
- **재색인 자동화**: 현재 수동 트리거뿐. 주기적 리프레시 스케줄 없음.

## 2. MediLaw 검색이 느린/타임아웃 원인 — 분석

팀원이 "WSL 환경이라 db 데이터가 작은데도 타임아웃되는 것 같다"고 했는데, 기록을 보면 **문서상 확정된 진짜 원인은 WSL 파일시스템이 아니었다.** 순서대로 정리한다.

### 2-1. 문서로 확정된 주원인 (이미 일부 해결됨)
- **임베딩 호출 순차 실행**: 검색 시 세그먼트마다 OpenAI 임베딩을 **하나씩 순차로** 부르고 있었음 → 약 **60초 지연**. 이건 **코퍼스 크기와 무관**(작은 db여도 느림 — 팀원 관찰과 일치). MediLaw에선 N→1 배치 + 캐시 + 병렬화로 고침(HMS rag fix).
- **판정 LLM 속도**: 그 수정 후에도 응답이 **50~130초** 걸리는데, 이건 **판정용 LLM(gpt-5.5) 자체 속도**가 지배적. 현재 남은 성능 과제로 표시돼 있음.

즉 "작은 db인데 타임아웃" = 데이터 크기 문제가 아니라 **(a) 임베딩 호출을 직렬로 돌린 것 + (b) LLM 생성 시간**이 핵심이었다.

### 2-2. WSL 관점에서 추가로 점검할 후보 (가설 — 확정 아님, 검증 권장)
팀원의 WSL 직감이 틀린 건 아니다. 위 주원인 외에 환경에서 지연을 더할 수 있는 것들:
- **DB가 `/mnt/c`(윈도우 마운트)에 있으면 느리다.** 실제로 `Desktop/MediLaw/.../medilaw.db`는 윈도우 경로에 있다. WSL에서 `/mnt/c`는 9p 프로토콜이라 디스크 I/O가 네이티브(`/home`, ext4) 대비 수십 배 느리다. sqlite-vec는 쿼리마다 벡터 블록을 디스크에서 읽으므로, 작은 DB라도 윈도우 마운트면 쿼리당 수 초가 나올 수 있다. → **DB를 `/home/user1`로 옮겨 보고 차이 측정**(가장 싸고 빠른 점검).
- **sqlite-vec는 brute-force KNN**(ANN 인덱스 없음). 21만 벡터×512를 매 쿼리 전수 스캔. 디스크 느림과 겹치면 체감 지연. (lawbot은 FAISS 인메모리라 이 디스크 이슈는 회피하지만, 1.36M brute-force는 별도로 1-1의 ANN 필요.)
- **async 안에서 동기 호출 블로킹**: HMS `documents.py`의 `/review`가 async 안에서 동기 함수를 직접 호출해 이벤트 루프를 막는 문제가 백엔드 검토에서 지적됨(여러 명 동시 시 멈춤). 이건 WSL과 무관하지만 "타임아웃처럼 보임"의 원인이 될 수 있음.
- **모델/확장 콜드스타트**: 첫 쿼리 때 sqlite-vec 확장 로드·모델 초기화 비용.

### 2-3. lawbot은 어떻게 다른가
lawbot은 FAISS를 **인메모리**로 올리고 retriever를 warm singleton으로 1회 로드하므로, `/mnt/c`·sqlite-vec brute-force·콜드스타트 같은 디스크성 지연을 구조적으로 회피한다. 단 위에서 말한 두 가지는 lawbot도 따로 챙겨야 한다: **(1) 136만 brute-force → ANN(1-1), (2) 검색 쿼리 임베딩을 캐시/병렬로**(이미 LRU 캐시 있음, 확인만).

## 3. 조례(자치법규)를 더 임베딩해야 하나 — 코퍼스 범위 결정

### 현재 상태
- 조례 **0건**(의도적 제외). 전체 코퍼스 390만 청크 중 조례가 63.5%(254만)인데 전부 뺐다 → 임베딩 비용 70%+ 절감.
- 근거: "전국법 커버는 전국 일반법으로 달성되지, 226개 지자체 조례 16만 건으로 달성되지 않는다." 조례는 노이즈가 크고 관할이 지역 한정.

### 세 가지 선택지
| 범위 | 추가 비용 | 권장도 | 비고 |
|---|---|---|---|
| **의료 관련 조례만** (39,659청크 / 3,496문서, ORD의 1.56%) | **~$0.2** | **권장** | MediLaw가 의료 도메인이라 가치 직접적, 거의 공짜 |
| 조례 제외 유지 | $0 | 기본값 | 일반 한국법 봇으로는 충분 |
| 전체 조례(254만) | **+$30~40** | 비권장 | 노이즈 큼. 굳이면 **별도 인덱스 + 관할 라우팅** 또는 law.go.kr 온디맨드 |

### 핵심 원칙 (전문가 의견 문서)
**노이즈의 원인은 "법 개수"가 아니라 "트리거 없는 무차별 검색"이다.** 그래서 해법은 코퍼스를 줄이는 게 아니라 **(a) 도메인 라우팅 (b) 주 근거/참고 인접 등급 분리**다. 같은 코퍼스도 라우팅·등급 분리가 있으면 노이즈가 통제된다. 조례를 넣더라도 격리 컬렉션 + 관할 라우팅으로 다루면 일반 검색을 오염시키지 않는다.

→ **권장 결론: 핵심 136만은 그대로 가고, 의료 조례 39,659개만 추가(~$0.2). 전체 조례는 넣지 않거나 별도 레이어로.**

## 4. 검색 성능을 더 좋게 하려면 — 구체적 방법

### 4-1. 하이브리드 (BM25 + RRF) — 가장 큰 레버
현재 retriever는 **dense(벡터) 단독**, BM25/RRF/rerank/FTS 전무. 법률 검색은 "제56조"·법령명·고시번호 같은 **정확 문자열 매칭**이 결정적이라 dense만으론 약하다.
- **한국어 토크나이저 함정**: SQLite FTS5 기본 unicode61은 형태소를 못 쪼갠다("의료광고를"로 색인되어 "의료광고"와 매칭 안 됨 → 0건). trigram은 부분 문자열은 잡지만 랭킹이 나빠지고, mecab-ko/khaiii는 C 확장 빌드 비용(NCP 단일서버). → **법령명·조문번호 중심의 명시적 키워드 사전 + 정규식 부스트가 ROI가 더 높다.**
- **반드시 먼저**: 위 1-4 #2(MIN_RETRIEVAL_SCORE 게이트)를 안 고치면 RRF 켜는 순간 전 질의가 차단된다.

### 4-2. 질의 정규화 (구어 → 법조문체)
"의료법 제56조 안 잡힘"의 근본 원인은 **구어와 법조문체의 어휘 격차**(코퍼스 노이즈 아님 — 깨끗한 의료 인덱스에서도 발생). 해법: 질의를 법 개념으로 재작성하거나 HyDE(가상의 답을 만들어 임베딩). 코퍼스 prune보다 ROI가 높다.

### 4-3. 인덱스 아키텍처 (1-1 재확인)
- **피해야 할 것**: 136만 무필터 단일 인덱스 + 사후 필터(과적출 k×5). doc_type=law가 6.8%처럼 선택도 낮은 필터는 정답이 top-40 밖으로 영구 탈락.
- **권장 A**: 도메인별 샤딩 + 라우팅(노이즈를 임베딩 공간에서 제거 → 사후 필터 recall 문제 없음).
- **권장 B**: 단일 HNSW(M=32, efSearch 64~128). 단 메모리 더 큼.

### 4-4. 이미 잘 되어 있어 건드리지 말 것
청크 parent/child 2단 구조(child=조문/판례섹션, parent=전문, parent_id 승격), 헤더를 벡터에 임베딩 — 이건 이미 강점. 손대지 말 것.

## 5. 만들 때 추가로 고려할 것 (아키텍처·운영 체크리스트)

- **NCP 서버 RAM을 먼저 확정.** 인덱스 ~3GB 상주 + LLM·OS 여유. 4vCPU/16GB(~₩140k/월)면 안전, 8GB면 양자화해도 풀로드 시 위험. 이게 인덱스 타입(Flat/IVF/HNSW)을 좌우한다.
- **제품 분리(Plan B 권장)**: lawbot = 순수 법률 두뇌 API(채팅 UI 없음), MediLaw = Node 제품(채팅/업로드/PDF 재생성/소비자 가드/계정)이 lawbot을 키로 호출. "법률 두뇌는 전부 lawbot, 화면·출력은 전부 MediLaw." 환각 로직 재구현 회피.
- **배포 형태**: Docker + NCP 단일서버, 공인 IP 먼저(도메인 나중, `<공인IP>.nip.io`로 Caddy HTTPS), GitHub 미사용·rsync로 코드/원천데이터(6GB)만 보내고 **벡터는 서버에서 빌드**(50GB+ 전송 회피).
- **`.env`의 stale `EMBED_DIM=1536` 한 줄 삭제**(import마다 경고, 기능 무해하지만 정리).
- **머지 글루**: 실제 남은 건 "벡터 조인" 한 개 — `embeddings.jsonl ⨝ chunks.jsonl on chunk_id → chunks_with_vectors.jsonl → faiss_index.build_index()`. `build_medical_index.py`에 패턴 이미 있음.
- **healthz가 Qdrant 레거시 프로브를 일부 남김** → FAISS 기준으로 정리.

## 6. lawbot.org(레퍼런스 베타) 대비 우리 위치

사용자가 말한 "인터넷의 베타 lawbot"은 우리가 만든 게 아니라 **외부 서비스 lawbot.org**다(2026-06-15 조사, `LAWBOT_ORG_RESEARCH.md`). 벤치마크로 비교:

| 항목 | lawbot.org (베타 v0.1.0) | 우리 lawbot |
|---|---|---|
| 포지셔닝 | "챗봇 아닌 법률 데이터 인프라", 인용 검증 중심 | 동일 지향 + `/v1/ask` 생성도 제공 |
| 임베딩 | Qwen3-Embedding-8B, 4096d, 셀프호스팅(비용 0) | OpenAI 3-small, 512d(비용 발생) |
| 벡터스토어 | Postgres+pgvector | FAISS |
| 검색 | **BM25+dense 하이브리드(RRF k=60)** | dense 단독(하이브리드 미구현) ← 따라가야 |
| 코퍼스 | MVP <1만 문서 | 136만 청크(전국 법령+판례) ← 우리가 훨씬 큼 |
| 제공형태 | REST + hosted + MCP | REST + MCP(구현됨) |
| 신뢰도 | 0~100 점수 + 신호등 + A/B/C/D | verify 골격 있음, **점수/신호등 UI는 TODO** |
| 가격 | 공개 없음(무료 데모 30req/min + 유료키 + 기업 문의) | 미정 |

배울 점: **한국 법률엔 BM25 하이브리드가 결정적**(정확 문자열 매칭), **신뢰점수·신호등 UX**, OpenAI 비용 커지면 셀프호스팅 임베딩 검토.

## 7. 더 필요한 정보 / 사용자(PM) 결정 사항

진행하려면 아래가 정해져야 한다.

1. **NCP 서버 스펙(특히 RAM)** — 인덱스 타입(Flat/IVF/HNSW)과 양자화 여부를 좌우. (4vCPU/16GB 권장)
2. **서비스 도메인 범위** — 의료만 소비할 건가, 일반 한국법 봇으로도 열 건가. MediLaw만 쓰면 전체 136만 인덱스의 즉시 가치는 낮고 의료 인덱스로 당장 출시 가능. 일반 봇까지면 전체 인덱스+ANN 필요.
3. **조례 포함 범위** — 의료 조례만(~$0.2, 권장) / 제외 유지 / 전체(+$30~40, 별도 레이어).
4. **실제 의료광고 PDF 샘플** — `/v1/ad-review` 품질을 측정할 실데이터. 이게 있어야 광고검토 정확도를 검증.
5. **골든셋 라벨링** — "구어 질의 → 정답 조문" 30~50쌍을 누가 법으로 검수할지. 검색 개선의 정량 기준.
6. **인증 헤더 통일** — 우리 `Authorization: Bearer lk_` vs lawbot.org `x-api-key`. 외부 사용자 일관성.
7. **도메인 보유 여부** — HTTPS 정식 인증서용(없으면 nip.io로 공인 IP HTTPS).
8. **스트리밍 정책** — lawbot `/v1/ask`는 SSE 있음. MediLaw가 타이핑 효과를 lawbot SSE로 받을지, 자체 OpenAI로 받고 출처만 `/v1/source-pack`로 받을지.

## 8. 권장 실행 순서

`★`는 임베딩과 무관하게 **지금 의료 인덱스로 병행 착수 가능**.

1. **머지 글루 → 전체 FAISS 빌드 + 무결성 게이트**(ntotal==meta·chunk_id 유일·L2≈1·B 제외). [반나절]
2. **인덱스 타입 결정·구축** — 136만이면 Flat 대신 **IVF 또는 HNSW**, NCP RAM 확정 후. recall/지연 측정. [1~2일]
3. ★ **골든셋 30~50쌍 작성** — 검색 개선의 기준선. [0.5~1일]
4. ★ **8대 버그 수정** — 특히 #1 as_of_date(법적 리스크)·#2 게이트(하이브리드 선행)·#4 정합성·#3 Firewall. [2~3일]
5. ★ **하이브리드(BM25+RRF) + 질의 정규화** — 게이트 수정 후. 전후 Recall@5/MRR@3 비교. [3~5일]
6. **(선택) 리랭킹** — 고위험 질의 조건부. [1~2일]
7. **운영 인프라** — 키별 쿼터·일일 상한, 메트릭, law.go.kr 캐시/타임아웃/CB, disclaimer·인증 통일. [2~3일]
8. **클라우드 실배포** — NCP 단일서버, 공인 IP HTTPS, healthz·키 발급 1회 통과, MCP 연결. [1~2일]
9. **(선택) 의료 조례 39,659개 추가**(~$0.2) + disclaimer 정합. [0.5일]

**핵심**: 3·4·5는 임베딩 완주를 기다릴 필요 없이 의료 인덱스에서 지금 샌드박스로 만들어 두면, 전체 인덱스가 나오는 순간 그대로 얹힌다.

대략 **2~4주면 production-ready** 도달 가능(설계 문서 추정과 일치).
