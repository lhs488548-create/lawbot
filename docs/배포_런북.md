# lawbot 배포 런북 (NCP 단일 서버 · Docker Compose)

현재 아키텍처(FAISS 전체 인덱스 + 하이브리드 + LLM 재작성 + Citation Firewall)를
네이버클라우드(NCP) 서버 1대에 올리는 단계별 가이드. 로컬 컨테이너로 이미 검증됨
(healthz 200·전 백엔드 ready·points 1,238,122). 코드·이미지는 배포 준비 완료 상태.

> 보안 원칙: 시크릿은 서버의 git-ignored `.env`에만. 절대 커밋·로그 금지.
> 자동화 스크립트: `scripts/deploy_ncp.sh` (로컬→서버 전송 + 기동).

---

## 0. 구성

```
  사용자 ──(HTTPS 443 / HTTP 80)──►  Caddy  ──►  api (FastAPI/uvicorn :8000)
                                              └─ FAISS 인덱스 + BM25 + parents.jsonl (파일 마운트)
```
- 벡터DB·Redis 컨테이너 없음(FAISS 파일 인덱스 + 인메모리 rate-limit).
- 데이터는 이미지에 굽지 않고 **호스트 아티팩트를 read-only 마운트**(`./artifacts:/app/artifacts:ro`).

## 1. 서버 준비 (NCP)
- 사양 권장: **vCPU 2+ / RAM 4GB+ / 디스크 30GB+**(아티팩트 ~12GB + 이미지 + 여유).
  - 서빙 RSS ~2.6GB(FAISS) + 첫 질의 시 인덱스 로드. RAM 4GB면 단일 인스턴스 OK, 여유는 8GB 권장.
- OS: Ubuntu 22.04 LTS 등. 공인 IP 1개.
- 방화벽(NCP ACG): **80, 443** 인바운드 허용(도메인 HTTPS). IP-HTTP만이면 80만. SSH(22)는 본인 IP만.
- Docker 설치:
  ```bash
  curl -fsSL https://get.docker.com | sh
  sudo usermod -aG docker $USER   # 재로그인 후 sudo 없이 docker 사용
  docker compose version
  ```

## 2. 코드 + 아티팩트 전송 (로컬 → 서버)
코드는 가볍고(소스만), 아티팩트는 ~12GB라 rsync로 보냅니다.

**자동(권장):** 로컬 WSL에서
```bash
cd /home/user1/lawbot
scripts/deploy_ncp.sh  user@<서버IP>          # 코드+아티팩트 전송, 원격 기동 안내까지
# 전송만:  scripts/deploy_ncp.sh user@IP --sync-only
```

**수동:**
```bash
# (a) 코드 (artifacts/.venv/.git 제외)
rsync -az --delete \
  --exclude artifacts --exclude .venv --exclude .git --exclude '*.db' \
  /home/user1/lawbot/  user@<서버IP>:/opt/lawbot/

# (b) 서빙 데이터만 전송 (full_index ~8.8GB + parents.jsonl ~3.3GB)
ssh user@<서버IP> 'mkdir -p /opt/lawbot/artifacts'
rsync -az --info=progress2 \
  /home/user1/lawbot/artifacts/full_index \
  /home/user1/lawbot/artifacts/parents.jsonl \
  user@<서버IP>:/opt/lawbot/artifacts/
```
> embeddings.jsonl·chunks.jsonl 등 원본은 서빙에 불필요 — 전송하지 않습니다.

## 3. .env 작성 (서버에서)
```bash
cd /opt/lawbot
cp .env.example .env
nano .env
```
필수/권장 값:
```
OPENAI_API_KEY=sk-...        # 필수. OpenAI 월 한도 설정 권장
LAWBOT_INDEX=full
LAWBOT_HYBRID=1
LAWBOT_REWRITE=1
LAWBOT_DEMO=1                # 무로그인 공개 데모. 키 전용 운영이면 0
GEN_MODEL=gpt-5-mini
LAW_OC=                      # law.go.kr OC 있으면 입력(firewall 현행 대조 강화)
# 도메인 분기 ↓ (4장)
```

## 4. 도메인 있음 / 없음 분기
**A. 도메인 있음 (자동 HTTPS, 권장)**
- DNS A 레코드를 서버 공인 IP로 지정.
- `.env`:
  ```
  DOMAIN=api.yourdomain.com
  ACME_EMAIL=you@example.com
  ```
- Caddy가 Let's Encrypt 인증서를 자동 발급·갱신(80/443 필요).

**B. 도메인 없음 (공인 IP·HTTP)**
- `.env`:
  ```
  DOMAIN=:80
  ```
  → Caddy가 80포트로 평문 HTTP 서비스(인증서 없음). 80만 열면 됨.
- 데모·테스트엔 충분. 공개 운영은 도메인+HTTPS 권장(키·문안 평문 전송 방지).

## 5. 기동
```bash
cd /opt/lawbot
docker compose config        # 렌더링 검증(에러 없어야 함)
docker compose up -d --build # 이미지 빌드 + 기동
docker compose ps            # caddy, api 가 healthy 인지
```
> 첫 기동 시 api 이미지 빌드(deps 설치) 수분 소요. 헬스체크가 healthy 될 때까지 대기.

## 6. 검증
```bash
# 도메인:    curl -fsS https://api.yourdomain.com/healthz
# IP-HTTP:   curl -fsS http://<서버IP>/healthz
```
기대: `{"ok":true,"points":1238122,"backends":{...,"faiss":true}}` (전 백엔드 true).
브라우저로 `https://도메인/chat`(또는 `http://IP/chat`) → 데모 페이지.

## 7. 관리자 키 발급 (키 전용 운영 시)
데모(LAWBOT_DEMO=1)는 키 없이 동작. 키 기반 테넌트 운영을 하려면:
```bash
docker compose exec api python -m api.auth     # 최초 admin 키 1회 출력 — 안전히 보관
# 이후 테넌트 키 발급:
curl -fsS -X POST https://도메인/v1/keys -H "Authorization: Bearer $ADMIN" \
  -H "Content-Type: application/json" -d '{"tenant":"acme","tier":"pro","rate":"120/minute"}'
```

## 8. 운영
- **백업**: `api_data`(키 DB), `caddy_data`(인증서) 도커 볼륨. 아티팩트는 로컬에서 재전송 가능.
- **갱신 배포**: 로컬에서 코드 수정 → `scripts/deploy_ncp.sh user@IP` 재실행 → 서버에서 `docker compose up -d --build`.
- **로그**: `docker compose logs -f api` / `... caddy`.
- **비용**: NCP 서버 월 요금 + OpenAI 사용량(임베딩은 인덱스에 이미 반영됨; 운영 비용은 ask/rewrite의 gpt-5-mini 호출 + 검색 임베딩). 재작성 캐시·law.go.kr 캐시로 반복 비용 절감.
- **데모 끄기**: 공개 남용 우려 시 `.env`에서 `LAWBOT_DEMO=0` → 키 전용.

## 9. 트러블슈팅
| 증상 | 원인/해결 |
|---|---|
| `docker compose config`가 OPENAI_API_KEY 오류 | `.env` 비었음 → 채우기 |
| healthz `faiss:false`·points null | 아티팩트 마운트 누락 — `/opt/lawbot/artifacts/full_index/index.faiss` 존재 확인 |
| api 컨테이너 재시작 반복 | `docker compose logs api`로 스택트레이스 확인(의존성·경로) |
| Caddy 인증서 실패 | 80/443 미개방 또는 DNS 미지정 — ACG·A레코드 확인, `logs caddy` |
| 첫 질의가 매우 느림 | 첫 호출 시 FAISS 인덱스 로드(수십 초~분). 이후 빠름 |
| OOM | RAM 부족 — 인스턴스 RAM↑(8GB 권장) 또는 데모 동시성 제한 |
