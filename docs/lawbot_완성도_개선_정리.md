# lawbot 완성도 개선 — 검증된 실행 정리 (lawbot 전용)

목표: lawbot을 **범용 한국법 grounding API**로 정식화한다 — 전체 코퍼스 임베딩 + 전체 FAISS 인덱스 + 최적화된 RAG 검색 + MCP + 클라우드 배포(API키) + 깔끔한 페이지.

출처: `lawbot_코퍼스_아키텍처_토론결과.md`(2026-06-26, 전문가 토론 + 라이브 파일 검증)에서 **실제 파일로 확인됐고 바로 행동에 옮길 수 있는 부분만** 추렸다. 추측·이미 반증된 주장은 부록으로 분리했다.

---

## 1. 큰 그림 — 지금 어디까지 왔고, 무엇이 남았나

**임베딩(재료)**: 코어 소스 `chunks.jsonl`은 1,357,018청크 = 법령 274,039 + 행정규칙 395,974 + 판례 687,005. 조례는 의도적으로 제외돼 0건(전체 390만에서 조례 254만을 뺀 값과 정확히 일치). 현재 약 42% 임베딩 완료(`embeddings.jsonl`).

**인덱스(현 상태)**: 지금 빌드돼 있는 FAISS 인덱스는 **의료 서브셋 하나뿐**(14,070청크). `config.FAISS_INDEX`가 이걸 가리키고 retriever도 이것만 로드한다. **lawbot 전체(한국법 전반)를 답하려면 전체 인덱스가 필요한데, 그건 아직 빌드된 적이 없다.** 그래서 임베딩 완주는 범용 lawbot의 실질적 선행조건이다. (의료 인덱스는 지금 유일하게 살아있는 인덱스라, 검색 알고리즘 개발·테스트용 샌드박스로만 활용한다.)

**핵심 통찰**: "구어·캐주얼 질의에서 정답 조문이 안 잡힌다"는 증상의 원인은 코퍼스 노이즈가 아니라 **검색 알고리즘**이다. 조례가 0건인 깨끗한 인덱스에 직접 질의해도 정답 조문(예: 의료법 제56조)이 top-50 밖으로 밀린다 — 구어체 질의와 법조문체 사이의 어휘격차를 밀집벡터 단독으로 못 메우기 때문이다. **코퍼스를 더 깎는 게 아니라 하이브리드 검색·질의 정규화가 처방이다.**

**따라서 "임베딩만 끝나면 최적화된 lawbot이 나온다"는 아니다.** 임베딩은 재료고, 그 위에 ① 전체 인덱스 빌드 ② 검색 알고리즘 최적화 ③ 제품화가 얹혀야 완성된다. 다행히 셋 다 이미 있는 코드 위에 얹는 작업이라 새로 짓는 수준은 아니다. 특히 ②는 임베딩 완주를 안 기다리고 의료 인덱스로 지금 착수할 수 있다.

## 2. 지금 당장 고쳐야 할 버그·리스크 (전부 lawbot 파일로 확인됨)

### 2-1. 현행 기준일(as_of_date) 게이트가 죽어 있다 — 법적 리스크 (최우선)

인덱스 청크의 **10.6%가 시행일이 오늘 이후**인 미래시행 조문이다(약사법 285건, 정보통신망법 202건, 건강보험법 177건 등). 그런데 `api/main.py`·`rag.ask`·`ad_review.review` 모두 `as_of_date` 기본값이 `None`이라 현행 필터가 한 번도 안 걸린다. 결과적으로 **아직 시행 안 된 개정본을 "현행"인 양 검색·인용**한다. 법령 grounding API에서 시행 전 조문을 현행으로 인용하면 오답이자 신뢰 리스크다. (구체 예: 의료법 제56조 본문 자체가 effective_from=2026-04-07이고 56조 청크는 인덱스에 1건뿐.)

조치: 호출 기본값을 서버 today로 깔거나, 미래시행 청크를 응답에서 "시행예정"으로 분리 라벨링. 시행일 도래 시 자동으로 "현행" 승격되는 메커니즘도 필요(지금 없음).

### 2-2. MIN_RETRIEVAL_SCORE=0.20 게이트 재점검 (높음)

두 문제가 겹친다. (a) 현재 dense-only 경로에서도 어휘격차로 구어 질의의 top_score가 0.20을 밑돌면 `rag.ask`가 모델 호출 없이 "근거 불충분"을 반환할 수 있다. (b) 나중에 RRF 하이브리드를 붙이면 RRF 점수는 최댓값이 1/(60+1)≈0.016이라 0.20을 절대 못 넘어 **모든 질의가 게이트에 걸린다.**

조치: 하이브리드 도입 전에 게이트를 **벡터 원점수 기준으로만** 평가하거나, RRF는 랭킹에만 쓰고 별도 임계를 둔다. 골든셋으로 실제 top_score 분포를 찍어 0.20 적정성을 재보정한다. (`ad_review`는 게이트 없이 항상 judge를 호출 — 경로별 동작이 갈리므로 일관화 대상.)

### 2-3. 메타설명문이 모든 청크 앞에 박혀 있다 — BM25·dense 양쪽 오염 (중간)

청크가 "이 조문은 '…'에 관한 사항을 규정한다"는 정형문구로 시작하는데, 이게 거의 모든 청크에 반복된다. 짧은 질의와의 코사인을 희석하고, BM25를 붙이면 이 상투구가 공통 출현해 IDF를 무너뜨린다.

조치: BM25 색인은 메타문을 빼고 **조문 본문·조문제목·법령명만** 색인(선택 아니라 필수 전처리). dense 희석은 헤더 포함/미포함 A/B로 측정.

### 2-4. law_kind 필터가 판례에서는 사건종류라 silent-miss (중간)

판례 law_kind는 `형사`·`일반행정` 두 값뿐(`민사`·`가사` 없음). 그런데 retriever `ALLOWED_FILTER_KEYS`에 law_kind가 살아 있어 사용자가 `{law_kind:"민사"}`로 거르면 **결과 0건이 조용히** 나온다. 행정규칙도 "규칙"이 "고시"로 라벨되는 오분류 존재.

조치: 판례는 별도 `case_category`로 빼거나 판례에 law_kind 필터 비허용. 법령류 law_kind는 규칙명 접미사로 정규화.

### 2-5. Citation Firewall 1차 게이트가 죽어 있다 (높음)

`verify.py:283`이 retriever에서 제거된 `get_qdrant_client`를 import해(retriever에 정의 0건) 1차 DB 실재확인이 항상 None 폴백으로 죽어 있다. 2차 문자대조는 살아 있어 명백한 환각은 폐기되지만, 1차가 없으면 폐지·미존재 조문을 "정확히 인용한" 환각이 통과할 수 있다.

조치: FAISS 시대 대체로 metas·parents.jsonl 조회 또는 source_url·사건번호 인덱싱으로 1차 실재확인 재구현.

### 2-6. ad_review 스코핑이 doc_type=law 고정이라 고시·행정규칙을 놓친다 (중간)

`ad_review.py`의 scoped 패스가 `flt={"doc_type":"law"}`로 고정이고 admrule은 unscoped k=2 보충에만 의존. 실증·고지·표시기준 같은 **판단 근거 상당수가 admrule에 있는데** k=2로는 회수가 빈약하다. 이미 인덱싱된 admrule을 검색단에서 구조적으로 과소회수하는 버그.

조치: scoped 패스에 admrule을 정당한 분량 포함하거나 law/admrule 2-패스로 각각 충분한 k.

### 2-7. 벌크 임베딩 경로에 trust_grade=B 필터가 없다 (중간)

medical 빌드 경로는 trust_grade=B(이미지·라벨만, 본문 없음)를 제외하지만 `embed_waves`엔 그 필터가 없다. 이미 임베딩된 구간의 약 16%가 별표·별지 라벨 청크(마지막 청크가 "별지0001"인 게 증거).

조치: `embed_waves` 스트림에 `trust_grade != B` 필터 추가, 인덱스 빌드 단계에도 이중 게이트. 빈 청크 노이즈와 비용 낭비를 동시 절감.

### 2-8. 인덱스–메타 정합성 검증이 없다 — 무음 오염 위험 (높음)

retriever는 `index.ntotal == len(metas)`와 row-id 정렬을 신뢰만 한다. 빌드 시 행 정렬·차원·중복 chunk_id 검증 단계가 없다. 전체 인덱스 빌드의 "얇은 조인(embeddings.jsonl ⨝ chunks.jsonl)"에서 정렬이 어긋나면 **모든 인용이 조용히 엉뚱한 출처를 가리키는** 최악의 무음 오염이 난다.

조치: 빌드 산출물에 `ntotal==len(meta)` · chunk_id 유일성 · 벡터 L2노름≈1 체크섬 게이트를 필수로.

## 3. 검색 품질의 진짜 해법 (임베딩과 독립 — 지금 착수 가능)

목표 파이프라인: 질의 → **도메인 라우팅으로 스코프 축소 → 구어 표현을 법률 개념으로 재작성(질의 정규화) → BM25와 dense 후보 생성 후 RRF 융합 → 고위험 질의에 한해 cross-encoder 재정렬 → as_of_date 현행 필터 → parent 승격 인용.**

**하이브리드(BM25+RRF)** — 현재 retriever에 BM25·RRF·리랭커·FTS5가 0건이다. 법률 텍스트는 "제56조"·법령명·고시번호 같은 정확 문자열 매칭이 결정적이라 밀집벡터만으론 약하다. 한국어 함정 둘: SQLite FTS5 기본 unicode61 토크나이저는 한국어 형태소를 못 나눈다("의료광고를"은 색인돼도 "의료광고"로는 매칭 0). trigram 토크나이저로 부분문자열 매칭을 확보하거나(랭킹 의미 일부 손상), mecab-ko/khaiii 커스텀 토크나이저(NCP 단일서버 C 확장 빌드 비용)를 쓴다. 법령명·조문번호 위주라면 **명시적 키워드 사전 + 정규식 부스트**가 가성비가 나을 수 있다. 그리고 2-2 게이트를 먼저 안 고치면 RRF 붙이는 순간 전건 "근거 불충분"이 된다.

**질의 정규화** — 어휘격차가 근본 원인이라 질의측 처리가 코퍼스 prune보다 직접적이다. 구어 질의를 법률 개념으로 재작성하거나 HyDE식 가설답변 임베딩을 쓴다.

**리랭킹(조건부)** — cross-encoder는 GPU 없는 NCP 단일서버에서 top-50 재정렬에 쿼리당 수백 ms가 붙는다. 무조건이 아니라 결론이 조문 하나에 달린 고위험 질의에 한정하고, 1차 점수가 애매할 때만 발동하는 게이트형으로.

청킹·부모문서 구조는 이미 우수(child=조문/판례섹션, parent=전문, parent_id 승격, 헤더가 벡터에 포함). 손댈 필요 없음.

## 4. 코퍼스 keep/drop 원칙 (전체 코퍼스 기준)

자르는 축은 세 가지의 곱: **위계**(법률 > 대통령령 > 부령 > 고시·훈령·예규 > 조례), **관할**(전국이냐 지방이냐), **실체성**(대외 권리·의무 규율이냐 내부행정이냐). 실측 분포 — 전체 390만 중 국가법령 6.8%, 조례 63.5%, 판례 17.2%, 행정규칙 9.9%.

**국가법령(법률·시행령·시행규칙)**: 전량 keep, 최상위. 모법에 시행령·시행규칙을 폴더 단위로 묶어 통째로 넣는 방식이 정답. lawbot 코어는 이미 전체 국가법령을 포함한다.

**행정규칙(고시·훈령·예규)**: 회색지대. 단일 휴리스틱 — "**이 규칙이 폐지되면 일반 수범자(기업·개인)의 행위규범이 바뀌는가**". 바뀌면 keep(실증규정, 표시·기재 기준, 보고·고지 의무, 평가 절차, 인증 기준 등), 공무원 내부 절차만 바뀌면 drop(직제·정원·위임전결·복무·계약·입찰·회계·예규). **명칭이 아니라 내용의 대외 규범성**으로 판단.

**조례(자치법규)**: 기본 drop, 이미 코어 제외. 단 "버린다"와 "격리한다"는 다른 결정이다. 옥외광고물·지역 인허가처럼 조례가 1차 규범인 영역이 있다. **단순 삭제가 아니라 jurisdiction 메타를 가진 별도 컬렉션으로 격리하고, 질의에 지자체명·조례 의도가 감지될 때만 라우팅.** 백업본 `chunks_full.jsonl`에 조례 254만 청크가 청킹된 채 보존돼 있어, 필요해지면 별도 인덱스로 색인하는 게 law.go.kr 온디맨드보다 일관성·지연 면에서 낫다. **조례 포함 여부는 lawbot 커버리지 범위에 대한 사용자 결정 사항**(비용 3~4배, 8장 참조).

**판례**: 좁게 정확매칭. 비용·노이즈 최대 진원. 대법원 사건 중 참조조문이 화이트리스트 법령을 정확 인용한 것만 keep(현재 622건 → 3,314청크). 하급심·키워드 확장(Phase-2)은 평가셋으로 품질 측정 후 개방. "넓게 넣고 거른다"가 아니라 "좁게 정확매칭".

## 5. 인덱스·검색 아키텍처 (전체 136만 규모 기준)

**규모가 의료(1.4만)와 다르다는 게 핵심.** 의료 14,070엔 `IndexFlatIP`가 정답(recall 100%, 수 ms)이었지만, 전체 136만은 다르다.

**전체 인덱스에서 절대 피할 단 하나**: 136만(또는 조례 포함 390만)을 **무필터 단일 인덱스에 넣고 사후필터로 골라내기.** FAISS는 네이티브 필터가 없어 Python 사후필터로 처리하는데 과적출이 k×5=40개뿐이라, 선택도 낮은 필터(예: doc_type=law=6.8%)를 걸면 정답이 후보창 밖으로 떨어져 **영구 미회수**된다. 즉 큰 인덱스의 메타필터는 노이즈를 거르는 게 아니라 정답까지 떨군다.

**대신 두 길 중 하나.** (a) **분야별 인덱스로 쪼개고 라우팅**(권장) — 노이즈가 임베딩 공간에서 아예 제거돼 사후필터 recall 문제가 없다. lawbot은 도메인 고정이 아니므로 질의에서 분야를 감지해 해당 인덱스로 보내는 라우팅이 필요. (b) **단일 인덱스라면 IndexHNSWFlat(M=32, efSearch 64~128)** — 단 HNSW 메모리는 원본의 2~5배(낙관적 1.2~2배 아님). 136만 Flat만 해도 136만×512×4byte ≈ 2.8GB, HNSW면 더 — **NCP 단일서버 사양을 미리 잡아야 한다.** 전환 임계는 단일 인덱스가 100만 벡터를 넘고 p95 지연이 SLA를 깰 때.

**지연은 코퍼스 크기와 무관.** 병목은 질의 임베딩 OpenAI 왕복·rate limit, 그리고 ad_review가 claim마다 순차 검색 + 인용마다 law.go.kr 최대 12회 동기 호출(`_VERIFY_MAX_CITATIONS=12`)하는데 캐시·타임아웃·서킷브레이커가 없다는 점. 외부 API가 느리면 지연이 곱으로 쌓인다 — 캐시·타임아웃·서킷브레이커 추가 필요.

**메타필터·시점검색은 이미 구현돼 있다**(doc_type·jurisdiction·law_kind·effective_from·parent_id + as_of_date). 위 라우팅/분리와 함께 보조로 쓴다.

## 6. 운영·빌드에서 빠진 것 (서비스급 필수)

- **회귀 골든셋의 정답 관리·동결.** "구어 질의 → 정답 조문" 30~50쌍을 만들되, 누가 법률 검수로 라벨링하고 법령 개정 시 어떻게 버저닝하는지 필요. 정답이 미래시행본으로 갱신되므로 **골든셋에 코퍼스 스냅샷 해시를 묶어** 핀고정해야 회귀 측정이 의미를 갖는다. 이건 이후 모든 검색 개선 판단의 전제.
- **상시 운영 지표.** 게이트 발동율, 인용 폐기율, as_of_date 미적용 비율, 쿼리 임베딩 캐시 적중률을 상시로 찍어야 품질 회귀를 사후에 안다.
- **개정 동기화 트리거·증분 단위 정의.** config의 LAW_OC·LAW_API_BASE는 있으나 "화이트리스트 폴더만 증분 재색인"이 무엇을 신호로 언제 도는지 명세가 없다. 시행일 도래 시 미래시행 청크를 현행 승격하는 메커니즘과 묶어야(2-1 연결).
- **재개 안정화.** `_load_done`의 5.6GB 풀스캔을 wave 완료마커나 SQLite done-set으로 교체해 재개가 갈수록 느려지는 걸 막는다. cancelling 비정상 종료를 백오프 재시도로 흡수. 대량 배치는 반드시 웨이브(≤7,600청크·≤2.6M토큰)로 — 과거 27샤드 실패는 단일 배치 5만 입력 한도 초과가 원인.
- **결합 글루(좁은 잔여작업).** "글루가 아예 없다"는 stale 서술이고 `build_medical_index.py`가 글루 패턴을 이미 구현했다. 전체 인덱스의 진짜 잔여는 **이미 과금한 웨이브 벡터를 버리지 않고 잇는 조인**(embeddings.jsonl ⨝ chunks.jsonl on chunk_id → {chunk_id,doc_id,parent_id,text,payload,vector} → faiss_index.build_index). build_index가 이 포맷을 이미 소비하므로 얇은 조인 하나면 됨.
- **제품 정합성.** `config.ANSWER_DISCLAIMER`가 "전국 자치법규" 포함을 광고하는데 코어가 조례를 제외하므로 허위 커버리지. "조례는 별도 레이어로 추후 제공"으로 고치거나 조례 레이어를 실제로 붙인다.
- **런타임 비용가드.** 현 가드(DEMO_MAX_CHUNKS·Batch 50%·캐시)는 전부 빌드시 가드. API키 단위 호출쿼터·일일 토큰상한이 없다. ad_review 한 건이 claim 다수 × (임베딩+검색+law.go.kr 12회)를 태우므로 한 키가 비용을 폭주시킬 수 있다.

## 7. 제품화 — 이미 있는 것 / 남은 것

**이미 있음**: `/v1/ask`(SSE 스트리밍)·`/v1/ad-review`·검색·검증 파이프라인, API키 인증(`api/keys.py`·`lawbot_keys.db`), Docker/배포설정, MCP 서버(`mcp_server/`, 도구 ask·search·verify·review), 깔끔한 페이지(`web/chat.html`, SSE·인용배지·diff뷰).

**남은 것**: 전체 인덱스 연결(retriever를 전체 인덱스로 지정), 위 검색 최적화·버그픽스, 디스클레이머 카피 정합성, 클라우드 배포 실빌드·헬스체크 1회 확인, API키 발급 운영 절차. healthz가 레거시 Qdrant 프로브면 FAISS로 정리.

## 8. 실행 순서

1. **임베딩 완주.** 현재 약 42%, Batch 24h 윈도에 묶여 며칠~1주. 조례까지 갈지는 별도 결정(비용 3~4배, 커버리지 vs 비용).
2. **전체 FAISS 인덱스 빌드 + 무결성 게이트**(ntotal==meta·chunk_id 유일성·L2노름). 벌크 trust_grade=B 필터도 이때 적용(2-7).
3. **검색 최적화(임베딩 안 기다리고 지금 착수 가능, 의료 인덱스로 테스트).** 골든셋 구축 → 버그픽스(2장) → 게이트 재설계 → 하이브리드(BM25+RRF) + 질의 정규화 → before/after Recall@5·MRR@3 검증.
4. **인덱스 아키텍처 확정**: 분야분할 라우팅(권장) 또는 HNSW. NCP 서버 메모리 사양 확정(5장).
5. **배포·운영**: retriever 전체 인덱스 연결, 클라우드 배포, API키 발급, 상시 지표·런타임 비용가드, 개정 동기화 명세.
6. **(선택) 조례 격리 레이어** + 디스클레이머 정합성.

핵심: **3번은 1·2를 안 기다리고 병행 가능**하다. 임베딩이 도는 동안 의료 인덱스를 샌드박스 삼아 하이브리드·게이트·버그픽스를 만들어두면, 전체 인덱스가 나오는 순간 그대로 얹힌다.

## 9. 비용 감각

임베딩은 content-hash 캐시 + Batch 50%로 낮다. 비조례 코어 전량(136만·약 9.9억 토큰)이 1회 약 $9.91, 42% 끝났으니 잔여 약 $5~6대(단 남은 청크가 대부분 판례라 전문이 길어 과소평가 가능 — tiktoken 실측 권장). **조례까지 가면 $30~40대로 3~4배**(조례는 코어와 텍스트가 거의 안 겹쳐 캐시 히트 5% 미만, 사실상 새 임베딩 1회). 즉 조례 제외는 전체 비용의 70%+ 절감 결정. 인덱스 메모리는 5장 참조(전체 Flat ≈ 2.8GB, HNSW면 더).

빠진 런타임 비용가드: 웨이브 실행 중 누적 토큰·비용 실시간 집계 + 상한 초과 자동정지, 일일 토큰 상한, 재색인 승인 절차.

## 부록 — 반증되어 채택하지 않은 주장 (혼동 방지)

- "조례까지 다 임베딩했다" → 아님. 코어에 조례 0건.
- "정답 조문 미포착의 원인은 코퍼스 노이즈" → 반증됨. 깨끗한 인덱스에서도 미포착. 원인은 어휘격차(검색 알고리즘).
- "TOTAL=1,357,018 분모가 틀렸으니 교정" → 아님. 비조례 코퍼스의 정확한 청크 수. "(비조례 기준)" 라벨링만 하면 됨.
- "embed_batch.py 차원 불일치는 무음 오색인" → 아님. `_validate_dim`이 1536≠512에서 즉시 ValueError로 하드 크래시. 그래도 dimensions=512 명시·캐시 정제는 값싸고 옳음.
- "결합 글루가 아예 없다" → stale. build_medical_index.py가 글루 구현됨. 잔여는 얇은 조인뿐.
- "특정 행정규칙 하나가 노이즈 대표" → 정렬상 맨 앞 헤드 아티팩트(전체의 0.1%). 노이즈 본질은 한 덩어리가 아니라 의료무관 2만+ 행정규칙·전분야 판례의 광범위 분산.
